Fix ACL denial classification to show red instead of amber

**Problem:** When a capability is revoked (e.g., relay:publish), the topology nodes flash amber instead of red. The isAmber logic checks for "relay" keyword without distinguishing ACL denials from infrastructure errors.

**Root cause:** The isAmber check at lines 148-154 catches both `"error: relay timeout"` (infrastructure, should be amber) and `"denied: relay:publish"` (ACL denial, should be red) because both contain the "relay" keyword.

**Solution:** Check if reason starts with `"denied:"` prefix BEFORE checking infrastructure keywords. This filters out denials and classifies infrastructure-only errors as amber.

## Message Format Distinction

| Scenario | Reason String Format |
|----------|----------------------|
| Infrastructure: no signer | `"error: no signer configured"` |
| Infrastructure: relay issue | `"error: relay timeout"` or similar |
| ACL Denial: relay revoked | `"denied: relay:publish"` ← Starts with `denied:` |
| ACL Denial: state write revoked | `"denied: state:write"` ← Starts with `denied:` |

Key insight: ACL denial messages ALWAYS start with `"denied:"` prefix (from runtime enforce.ts line 201). Infrastructure errors do NOT use this prefix.

```typescript
  reasonString.includes('relay') ||
  reasonString.includes('timeout') ||
  reasonString.includes('not wired') ||
  reasonString.includes('mock')
);
const isAmber = isOkFalse && isInfrastructureError;
```
