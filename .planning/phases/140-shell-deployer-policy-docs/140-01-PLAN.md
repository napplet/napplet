<!-- Citation targets the doc MUST reference -->

Normative spec cited as "NUB-CONNECT":
  - Manifest tag shape: `["connect", "<origin>"]`
  - Origin format: scheme whitelist (https/wss/http/ws), lowercase host, Punycode IDN, no wildcards, no default ports, no path/query/fragment, no trailing slash
  - aggregateHash fold: `connect:origins` synthetic xTag, lowercase → ASCII sort → `\n`-join (no trailing) → UTF-8 → SHA-256 → lowercase hex
  - Runtime API: `window.napplet.connect: { readonly granted: boolean; readonly origins: readonly string[] }` — NEVER `undefined`
  - Capability strings: `nub:connect`, `connect:scheme:http`, `connect:scheme:ws`
  - Discovery meta tag: `<meta name="napplet-connect-granted" content="<space-separated-origins>">`
  - Grant key composite: `(dTag, aggregateHash)`

Class-2 posture cited as "NUB-CLASS-2.md":
  - CSP shape: `connect-src <space-separated-list-of-granted-origins>`
  - Residual meta-CSP MUST refuse-to-serve (intersection of `<granted>` and `'none'` silently reduces to `'none'`)
  - Grant persistence: composite key, diff UI on re-prompt, strict-superset extended-key hygiene
  - Revocation: DENIED-not-deleted; effective at next frame creation; no in-session class re-assignment
  - Denied napplets served as Class-1

Class-1 posture cited as "NUB-CLASS-1.md":
  - CSP: `connect-src 'none'`
  - Residual meta-CSP is harmless (intersection of `'none'` and `'none'` is `'none'`)

    ## Required structure (13 H2 sections total)

    ### Top-of-file (before first H2)
    - H1 title: `# Shell Connect Policy Checklist`
    - Blockquote banner (2 lines): "Shell-implementer guide for v0.29.0+ napplet hosts implementing NUB-CONNECT." / "Normative wire shape and MUST/SHOULD language live in NUB-CONNECT (napplet/nubs); this document is a deployment checklist."

    ### `## Status`
    - Self-label: "This is a non-normative implementer's guide."
    - Normative source: "The normative spec is **NUB-CONNECT** in the `napplet/nubs` repo."
    - Posture delegation note: "Class-2 posture details — CSP shape, consent-flow MUSTs, residual-meta-CSP refuse-to-serve, revocation UX — are owned by `NUB-CLASS-2.md` in the same repo."
    - One-sentence bridge sentence noting that SHELL-RESOURCE-POLICY.md covers the sibling read-only NUB.

    ### `## Why this exists` (POLICY-01 — motivation for the companion doc's existence)
    - Frame: "NUB-CONNECT hands the napplet a direct network socket to user-approved origins. That grant bypasses every shell-side filter NUB-RESOURCE provides (MIME sniffing, private-IP block, redirect re-validation, SVG rasterization). The shell's defense-in-depth degrades to: (a) getting informed consent right, (b) getting the CSP header right, (c) refusing to serve a class of broken manifests that would silently suppress the grant."
    - State explicitly: "A shell that implements NUB-CONNECT incorrectly does not fail loudly — it fails silently, with the user believing the napplet has network access it does not actually have, or with the napplet reaching origins the user did not approve."
    - Close with one sentence pointing deployers at the audit checklist at the bottom.

    ### `## HTTP Responder Precondition (MUST)` (POLICY-02)
    - State: "Shell MUST own the HTTP response headers for the napplet's HTML. NUB-CONNECT's CSP grant is delivered via the `Content-Security-Policy` HTTP response header; a shell that does not control the response cannot enforce this NUB."
    - Enumerate the four delivery modes from CONTEXT.md with per-mode pitfalls, each as a bulleted subsection under a `### Delivery Modes` H3:
      * **Direct serving** — shell controls headers fully; watch CDN/edge caches that MAY strip or override CSP headers; the grant varies per `(dTag, aggregateHash)` so caches keyed on URL alone MUST be invalidated on grant change.
      * **HTTP proxy** — the proxy MUST preserve or rewrite the shell's emitted CSP; intermediate proxies silently merging two origins' CSPs via intersection will break Class-2 grants exactly the way residual meta-CSP does.
      * **`blob:` URL with HTML rewrite** — `blob:` URLs have no HTTP response and therefore no HTTP CSP header; the shell is forced back to `<meta http-equiv="Content-Security-Policy">` for the grant, which intersects (not overrides) any residual meta-CSP in the napplet HTML. MUST strip residual meta-CSP before serving.
      * **`srcdoc=`** — `about:srcdoc` origin resolves `'self'` against the parent frame's origin rather than an opaque napplet origin; CSP directives that reference `'self'` have subtly different semantics. MUST either avoid `'self'` in `connect-src` (NUB-CONNECT never uses `'self'` anyway) or accept that the posture becomes parent-frame-scoped.
    - Close with: "Each mode MUST be validated end-to-end — with a real napplet, a real grant, and a real `fetch` to a granted origin — before the deployment is considered compliant."

    ### `## Residual Meta-CSP Scan (MUST)` (POLICY-03 — parser-based example + 5-fixture bundle)
    - State the rule: "Shells MUST refuse to serve any napplet with a NUB-CLASS-2 posture (manifest contains at least one `['connect', ...]` tag) whose HTML body contains a `<meta http-equiv=\"Content-Security-Policy\">` element. Browser CSP intersection silently reduces header `connect-src <granted>` AND meta `connect-src 'none'` (or ANY meta `connect-src`) to the narrower of the two — which in practice suppresses the grant with zero diagnostic."
    - State the Class-1 distinction: "For napplets with a NUB-CLASS-1 posture (no `connect` tags), residual meta-CSP is harmless per `NUB-CLASS-1.md` (intersection of `'none'` and `'none'` is still `'none'`). The scan MUST apply only to NUB-CLASS-2 napplets."
    - `### Why parser, not regex` — Subsection explaining: The residual meta-CSP scan runs server-side against arbitrary napplet-author HTML. Regex-based scanners miss attribute-reordering, case variations, single- vs double-quoted values, HTML comments that mask the element, and CDATA sections. The vite-plugin's `assertMetaIsFirstHeadChild` regex is acceptable for the build-time path — author controls that HTML shape — but shell-side scanning MUST use a real HTML parser (e.g., `parse5`, `htmlparser2`, or an equivalent WHATWG-compliant parser).
    - `### Parser-based example` — HTML code block showing pseudocode like:
      ```
      const document = parse5.parse(napplethtml);
      const metas = findAllElements(document, (el) =>
        el.tagName === 'meta' &&
        getAttrValue(el, 'http-equiv')?.toLowerCase() === 'content-security-policy'
      );
      if (metas.length > 0) {
        throw new RefuseToServe({
          reason: 'residual-meta-csp',
          element: serializeElement(metas[0]),
          remediation: 'napplet authors: remove <meta http-equiv="Content-Security-Policy"> from dist/index.html; CSP is now delivered via HTTP response headers.',
        });
      }
      ```
    - `### 5-Fixture Conformance Bundle` — Numbered list of 5 HTML fixtures a conformant scanner MUST reject. Each fixture is a code block.
      1. Standard attribute order: `<meta http-equiv="Content-Security-Policy" content="default-src 'self'">`
      2. Attribute reordering: `<meta content="default-src 'self'" http-equiv="Content-Security-Policy">`
      3. Inside an HTML comment (SHOULD also be caught — some templating systems leak commented meta through intermediate processors, and the parser's comment-handling must not silently discard):
         `<!-- staging fixture --> <meta http-equiv="Content-Security-Policy" content="...">`
         NOTE: The element is active (NOT inside the comment itself). This fixture tests that the scanner correctly parses HTML comments (does not get confused by a nearby `<!--`) and still detects the live meta after.
      4. Inside a `<![CDATA[...]]>` section (within `<svg>` or similar) — fixture demonstrates that a parser correctly suppresses CDATA content (so the meta inside CDATA is NOT active and MUST NOT cause refusal), proving that the scanner is parser-based and not a dumb text-grep:
         `<svg><![CDATA[<meta http-equiv="Content-Security-Policy" content="...">]]></svg>`
         (Expected behavior: NO refusal. This is the negative test — the bundle proves correct parsing.)
      5. Single-quoted attribute values + case variation: `<META HTTP-EQUIV='content-security-policy' CONTENT='default-src *'>`
    - State: "A conformant scanner MUST reject fixtures 1, 2, 3, and 5, and MUST NOT reject fixture 4. Shells SHOULD include all five fixtures in their conformance test suite."

    ### `## Mixed-Content Reality Check (MUST)` (POLICY-04)
    - State: "Browsers block `http:` and `ws:` fetches from napplets whose shell is served over `https:`, REGARDLESS of the emitted `connect-src` CSP value. This enforcement runs below the CSP layer and cannot be relaxed by the shell."
    - Enumerate the exceptions: `localhost`, `127.0.0.1`, `[::1]` are treated as secure contexts by browsers and can be fetched from HTTPS origins.
    - State: "A napplet declaring cleartext origins that get approved by the user will silently fail to fetch when served from an HTTPS shell. The shell MUST surface this reality in the consent prompt when cleartext origins are under consideration, and SHOULD log a deployment warning during serving."
    - Reference the build-tool's cleartext warning path (NUB-CONNECT Security Considerations) so deployers know the layer already emits one signal.

    ### `## Cleartext Policy (MUST)` (POLICY-05)
    - State: "Operator policy MAY refuse cleartext entirely. Shells refusing cleartext MUST advertise `shell.supports('connect:scheme:http') === false` and `shell.supports('connect:scheme:ws') === false` via the NIP-5D capability query API. Shells MUST NOT silently strip cleartext origins from a manifest — refuse-to-serve with a diagnostic is the correct behavior, not silent downgrade."
    - State: "Shells permitting cleartext MUST mark each cleartext origin visibly in the consent prompt with the confidentiality tradeoff: the user is approving unencrypted traffic with that origin, subject to mixed-content rules above."
    - State: "The shell-level capability advertisement is the authoritative signal; napplet-build tools inspect it and degrade gracefully when cleartext is refused, per NUB-CONNECT Graceful Degradation."

    ### `## Grant Persistence (MUST)` (POLICY-06)
    - State: "Grants MUST be keyed on the exact composite `(dTag, aggregateHash)`. Keying on `dTag` alone is a security bug — it allows a rebuilt napplet with a changed origin set (new `aggregateHash` via the `connect:origins` fold) to inherit the prior approval silently. See `NUB-CLASS-2.md`'s Grant Persistence Semantics section for the complete normative requirements."
    - State: "The `connect:origins` fold defined in NUB-CONNECT produces a digest that participates in `aggregateHash`; ANY change to the declared origin set — addition, removal, or reorder after normalization — produces a new `aggregateHash` and auto-invalidates the prior grant, triggering a fresh consent prompt on next load."
    - State: "Shells MAY extend the keying tuple (e.g., include author pubkey or network partition), but the extended key MUST be a strict superset of `(dTag, aggregateHash)` to remain portable across conformant shells."

    ### `## Revocation UX (MUST)` (POLICY-07)
    - State: "Shells MUST expose a user-facing revocation affordance for approved grants. Users who approved a grant MUST be able to revoke it without needing to uninstall or delete the napplet."
    - State: "Revoked grants MUST move to a DENIED state rather than being deleted. Shells retain historical knowledge — the user approved once, then revoked — which enables (a) re-approve paths that surface 'you previously approved these origins' context, and (b) audit UX showing past decisions."
    - State: "Revocation takes effect no later than the next time the affected `(dTag, aggregateHash)` is loaded. Shells MUST NOT emit revoked origins in `connect-src` for any napplet frame created after the revocation is acknowledged. In-session (mid-lifecycle) class re-assignment is out of scope — NUB-CLASS's at-most-one-envelope rule forbids it — but the next frame creation under the same key MUST see the denied posture."
    - Reference `NUB-CLASS-2.md`'s Revocation Timing security consideration for the normative source.

    ### `## Consent UX Language (MUST)` (POLICY-08)
    - State: "The consent prompt MUST capture the following semantic elements (exact wording is a shell-UX decision, but the MUSTs are load-bearing regardless of copy):"
    - Numbered list mirroring NUB-CONNECT Shell Consent Flow:
      1. Napplet name as declared in the NIP-5A manifest.
      2. Complete list of requested origins verbatim (NO summarization, NO truncation, NO first-presentation elision).
      3. Explicit statement that approval allows the napplet to **send AND receive** any data with the listed origins — the grant is not read-only, not shell-mediated, and not rate-limited.
      4. Explicit statement that the shell cannot see or filter post-grant traffic between the napplet and the listed origins.
      5. Visible marking of any cleartext (`http:` or `ws:`) origins alongside the confidentiality tradeoff.
    - State: "Shells MUST NOT use diminutive language ('just', 'only', 'simply') that understates the trust cost of approval. Write the prompt for an informed user; assume they can handle accurate language."
    - Include the exemplar phrasing: "An acceptable one-sentence framing: *'This napplet can talk with foo.com however it wants.'* Shells MAY vary the wording but MUST convey the full scope of the grant."

    ### `## Explicit N/A Items` (POLICY-09 — what NUB-CONNECT deliberately does NOT require shells to do)
    - Framing sentence: "The following items appear in NUB-RESOURCE's policy surface but are DELIBERATELY not required for NUB-CONNECT, because the browser enforces CSP transparently to the shell and the shell has no post-grant hook:"
    - Bulleted list:
      * **Private-IP block list** — NUB-RESOURCE validates against resolved IPs at DNS-resolution time. NUB-CONNECT cannot: browser CSP `connect-src` matches the URL literal, not the resolved IP, and DNS rebinding would bypass any shell-side origin-match check regardless. This is a NUB-RESOURCE-only concern; shells implementing both NUBs MUST enforce the block list for the RESOURCE path and document that the CONNECT path does not provide the same guarantee.
      * **MIME byte-sniffing** — shell does not see post-grant bytes.
      * **SVG rasterization caps** — shell does not see post-grant bytes.
      * **Redirect chain limits** — browser follows redirects below the shell's visibility.
      * **Size / rate caps on granted traffic** — browser enforces CSP transparently; no shell hook exists to impose per-request quotas without a service worker, which the napplet sandbox forbids.
    - Close with: "Deployers MUST NOT interpret the absence of these items as 'safer than NUB-RESOURCE'; the correct interpretation is 'weaker posture, narrower threat model, requires full trust in the user's consent decision.'"
