<objective>
Migrate shell to use service types from core and runtime, and wire services pass-through in hooks-adapter.

Purpose: After Plan 01 (ServiceDescriptor in core) and Plan 02 (ServiceHandler/ServiceRegistry in runtime),
the shell's local definitions of these types are redundant. This plan removes the local definitions,
imports from the canonical packages, updates ShellHooks to use the imported types, and adds the
services pass-through in hooks-adapter.ts so shell hosts can provide services that reach the runtime.

Output:
- packages/shell/src/types.ts — Remove local service type definitions, import from core/runtime
- packages/shell/src/index.ts — Re-export service types from core/runtime (backwards compatibility)
- packages/shell/src/hooks-adapter.ts — Pass services through to RuntimeHooks
</objective>

## 1. Shell types.ts changes

Current state (lines 218-287): ServiceDescriptor, ServiceHandler, ServiceRegistry are all defined locally.

Target state: Remove all three local interface definitions. Import them from core/runtime. ShellHooks.services
remains with the same external shape but references the imported ServiceRegistry.

Add imports at the top of types.ts:
```typescript
import type { ServiceDescriptor } from '@napplet/core';
import type { ServiceHandler, ServiceRegistry } from '@napplet/runtime';
```

Note: types.ts already has `import type { NostrEvent } from '@napplet/core';` — add ServiceDescriptor to that import.
Note: types.ts may or may not already import from @napplet/runtime — add the new import.

Remove these three blocks (lines 218-287):
- `export interface ServiceDescriptor { ... }` (lines 218-225)
- `export interface ServiceHandler { ... }` (lines 244-263)
- `export interface ServiceRegistry { ... }` (lines 285-287)

Also remove their JSDoc comments and section divider.

Re-export the imported types so that `import { ServiceDescriptor } from './types.js'` still works
within the shell package:
```typescript
// Re-export service types from canonical packages
export type { ServiceDescriptor, ServiceHandler, ServiceRegistry };
```

The ShellHooks.services field (line 335) stays as-is — it already references `ServiceRegistry` by name
and the type will now resolve to the imported one.

Current state (lines 38-41):
```typescript

  ServiceRegistry,
```

These re-exports should now come from the canonical packages for type consumers who import
directly from '@napplet/shell'. The current import from './types.js' will still work since
types.ts re-exports them, but for clarity the shell index should re-export from the canonical
source.

Change the service type exports to come from canonical packages:
```typescript
// Service types (re-exported from canonical packages for backwards compatibility)

export type { ServiceHandler, ServiceRegistry } from '@napplet/runtime';
```

Remove ServiceDescriptor, ServiceHandler, ServiceRegistry from the existing `from './types.js'` export block.

## 3. Hooks-adapter services pass-through

Current state: The adaptHooks() function returns a RuntimeHooks object (line 315-330) that does NOT
include a `services` field. The shell's `ShellHooks.services` is never passed to the runtime.

Target state: Add `services: shellHooks.services` to the returned RuntimeHooks object.

In the "Assemble RuntimeHooks" section (line 315), add to the return object:

```typescript
  return {
    sendToNapplet,
    relayPool,
    cache,
    auth,
    config,
    hotkeys,
    crypto: cryptoHooks,
    aclPersistence,
    manifestPersistence,
    statePersistence,
    windowManager,
    relayConfig,
    dm,
    onAclCheck: shellHooks.onAclCheck,
    services: shellHooks.services,
  };
```

This is a simple pass-through. ShellHooks.services and RuntimeHooks.services have the same
ServiceRegistry type (both imported from @napplet/runtime after migration).
