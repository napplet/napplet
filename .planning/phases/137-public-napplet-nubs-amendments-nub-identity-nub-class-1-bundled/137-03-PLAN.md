    ````
    ### Error Codes

    The `identity.decrypt.error` envelope's `error` field takes exactly one of the 8 values in the `IdentityDecryptErrorCode` vocabulary. Each code names a distinct failure surface:

    ```typescript
    type IdentityDecryptErrorCode =
      | 'class-forbidden'
      | 'signer-denied'
      | 'signer-unavailable'
      | 'decrypt-failed'
      | 'malformed-wrap'
      | 'impersonation'
      | 'unsupported-encryption'
      | 'policy-denied';
    ```

    | Code | Failure surface |
    |------|-----------------|
    | `class-forbidden` | The napplet is not assigned `class: 1` per `NUB-CLASS-1.md`. `identity.decrypt` is unavailable from any other class. This is the shell's class-gating rejection; the shell MUST emit it before attempting any crypto operations. |
    | `signer-denied` | The user explicitly denied the shell's signer prompt (if one was raised). The napplet SHOULD treat this as a user-intentional outcome and not retry automatically. |
    | `signer-unavailable` | No signer is configured or the configured signer is unreachable. The shell could not attempt decryption because the private-key-holding component is absent. |
    | `decrypt-failed` | A crypto operation failed during decryption — wrong recipient, corrupted ciphertext, bad derivation, NIP-44 MAC mismatch, or any other cryptographic failure the shell detected.     | `malformed-wrap` | The outer wrap event failed structural validation — missing required fields, outer signature verification failed, or the wrap shape does not parse as NIP-04 / NIP-44 / NIP-17 after best-effort decoding. The shell MUST emit this BEFORE attempting seal decrypt; outer-signature verification is a precondition. |
    | `impersonation` | For NIP-17 flows, the shell detected `seal.pubkey !== rumor.pubkey`. The rumor is unsigned and is attacker-controllable; a mismatch between the seal's authenticated pubkey and the rumor's claimed pubkey indicates the sender is attempting to impersonate a different user. The shell MUST refuse delivery. |
    | `unsupported-encryption` | The event's shape does not match any of the three supported encryption modes (NIP-04, direct NIP-44, NIP-17). The shell MAY support additional modes via implementation-specific extension, but MUST return this code when no built-in mode applies. |
    | `policy-denied` | The shell's per-napplet or per-user policy refuses the request (rate limit, deny-list, prior CSP-violation strike against the napplet per `NUB-CLASS-1.md` Shell Responsibilities, etc.). The policy surface is shell-side UX concern; this code covers the protocol-visible refusal. |

    The `message?` field on the `.error` envelope is a human-readable string intended for diagnostic logging only. Programmatic dispatch MUST be driven by the `error` code. Shells MAY populate `message?` with implementation-specific detail (e.g., "current class is 2" for `class-forbidden` to aid debugging) but MUST NOT include other napplets' identifiers, other users' pubkeys, or environmental secrets — see `class-forbidden` details under Shell Behavior.
    ````

    ```
    ### Receive-Side Decrypt Surface (identity.decrypt)

    The `identity.decrypt` method exposes plaintext to a napplet and carries three distinct security concerns that this subsection addresses individually.

    **NIP-17/59 gift-wrap impersonation.** NIP-17 messages use an outer gift-wrap event (kind 1059) containing an encrypted seal event (kind 13) containing an unsigned rumor. The rumor is intentionally unsigned — NIP-59 sender-anonymity depends on the outer `created_at` being randomized and the inner rumor carrying no signature that would authenticate the sender to a passive observer. The authenticated sender is the seal's `pubkey` (the seal is signed). A hostile sender can produce a gift-wrap whose seal is signed by themselves but whose rumor claims a different `pubkey`. The shell MUSTs in Shell Behavior prevent this: outer signature verification (`malformed-wrap` on fail), seal-unwrap, seal-to-rumor pubkey equality check (`impersonation` on mismatch), and surfacing the authenticated `sender` separately from the `rumor` on the result envelope. Napplets MUST treat `rumor.pubkey` as untrusted and use `sender` as the authenticated sender identity.

    **NIP-07 extension content-script injection.** Browser extensions implementing NIP-07 (the original legacy signer-provider interface) frequently ship content scripts with `all_frames: true`, injecting `window.nostr` into every frame on the page — including sandboxed napplet iframes. On shells that do not enforce strict CSP, this injection succeeds and the napplet sees a `window.nostr` it was not supposed to see; under the NUB-CLASS-1 posture combined with a nonce-based `script-src` directive (`script-src 'nonce-<per-page-nonce>' 'self'`), the legacy `<script>`-tag injection path is blocked by the browser — the extension's content script cannot know the per-page nonce and the injected `<script>` element has no valid nonce attribute. Browsers fire a `securitypolicyviolation` event in this case (Chromium emits `violatedDirective: "script-src-elem"` on recent versions, `"script-src"` on older versions; both belong to the `script-src` directive family), which feeds into the `report-to` mechanism specified by `NUB-CLASS-1.md`. Together, NUB-CLASS-1's CSP posture and `NUB-CLASS-1.md`'s Shell Responsibilities (violation-report processing via `(dTag, aggregateHash)` correlation) give shells a mechanism to observe legacy-injection attempts against their napplets.

    **Extension main-world residual.** The `script-src` nonce does NOT block extension-originated JavaScript that runs in the page's main world via `chrome.scripting.executeScript({world:'MAIN'})` or equivalent Manifest V3 paths. Such injections bypass page CSP entirely by extension-API design — no `securitypolicyviolation` fires, no report is POSTed, and no page-side mechanism detects or prevents the injection. This NUB acknowledges the residual honestly; it does NOT claim `identity.decrypt` is safe from a hostile extension that reaches into the main world. The structural mitigation is `connect-src 'none'` (emitted by shells per `NUB-CLASS-1.md` § CSP Posture): any plaintext obtained inside a NUB-CLASS-1 frame — via `identity.decrypt`, via any other mechanism — cannot be exfiltrated over the network from within that frame because `connect-src 'none'` blocks fetch, XHR, WebSocket, EventSource, and sendBeacon uniformly. Plaintext is trapped inside the frame regardless of how it was obtained. This is the load-bearing reason `identity.decrypt` is gated to NUB-CLASS-1 napplets only: a NUB-CLASS-2 napplet (with user-approved direct-origin access per NUB-CONNECT) has a network egress path that defeats this structural mitigation.

    **Shim-side observability, not a trust boundary.** A napplet's shim SDK MAY short-circuit `window.napplet.identity.decrypt` locally when `window.napplet.class !== 1`, returning `class-forbidden` without crossing the postMessage boundary. This is a developer-experience affordance — it gives napplet authors an immediate, synchronous-feeling failure in dev environments — and is explicitly NOT a trust boundary. Napplets are untrusted in this model (the shim SDK is part of the untrusted surface; a hostile napplet can bypass it trivially). Shell enforcement is authoritative: the shell MUST reject the envelope regardless of whether the shim ran the local short-circuit. Treating the shim's behavior as evidence of compliance would be a security error.
    ```
