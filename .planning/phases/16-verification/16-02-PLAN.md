# Plan 16-02: Runtime Package Unit Tests for Message Dispatch

## Objective

Create isolated unit tests for @napplet/runtime that verify message dispatch logic (EVENT, REQ, CLOSE, COUNT, AUTH) runs correctly without a browser environment. These tests prove the protocol engine is truly browser-agnostic.

## Context

Phase 13 creates @napplet/runtime with the protocol engine extracted from shell-bridge.ts. TST-03 requires isolated unit tests for message dispatch that run without a browser. The runtime depends only on @napplet/core and @napplet/acl — no DOM APIs.

The tests must create a runtime instance via `createRuntime(hooks)` with mock RuntimeHooks, then send NIP-01 messages and verify correct behavior. This proves the runtime can be hosted outside a browser.

- `verifyEvent(event)` — Returns true for any well-formed event (or uses nostr-tools if available as devDep)
- `persistAcl(state)` / `loadAcl()` — In-memory storage (Map or simple object)

Structure:
```ts
import type { RuntimeHooks } from './types.js';

export interface MockRuntimeHooks {
  hooks: RuntimeHooks;
  sent: Array<{ windowId: string; message: unknown[] }>;
  published: unknown[];
  reset(): void;
}

export function createMockRuntimeHooks(overrides?: Partial<RuntimeHooks>): MockRuntimeHooks {
  const sent: Array<{ windowId: string; message: unknown[] }> = [];
  const published: unknown[] = [];

  const hooks: RuntimeHooks = {
    sendToNapplet(windowId: string, message: unknown[]) {
      sent.push({ windowId, message });
    },
    // ... other methods with sensible mock implementations
    ...overrides,
  };

  return {
    hooks,
    sent,
    published,
    reset() {
      sent.length = 0;
      published.length = 0;
    },
  };
}
```

The exact properties on RuntimeHooks depend on Phase 13's implementation. The executor must read the actual RuntimeHooks interface and implement mocks for every required method.

<action>
Create `packages/runtime/src/dispatch.test.ts` with tests for each NIP-01 verb:

**1. REQ verb tests:**
- Send `['REQ', 'sub-1', { kinds: [1], limit: 10 }]` from an authenticated napplet
- Verify runtime creates a subscription and sends matching events back via `sendToNapplet`
- Verify REQ with invalid filter (missing kinds) is handled gracefully
- Verify REQ from unauthenticated napplet is rejected or queued

**2. EVENT verb tests:**
- Send `['EVENT', { id: '...', kind: 1, content: 'hello', ... }]` from an authenticated napplet
- Verify event is delivered to other napplets with matching subscriptions
- Verify event with kind matching BusKind.SIGNER_REQUEST routes to signer handler
- Verify event with kind matching BusKind.INTER_PANE routes inter-pane

**3. CLOSE verb tests:**
- After REQ, send `['CLOSE', 'sub-1']`
- Verify subscription is removed and no further events are delivered for that subId
- Verify CLOSE for non-existent subscription does not crash

**4. AUTH verb tests:**
- Send AUTH challenge, verify napplet receives `['AUTH', '<challenge>']`
- Send AUTH response with valid signature, verify authentication succeeds
- Verify authenticated identity is registered in key registry

**5. COUNT verb tests (if supported):**
- Send `['COUNT', 'count-1', { kinds: [1] }]` from authenticated napplet
- Verify runtime responds with count or delegates to relay pool

**6. Capability enforcement tests:**
- Create runtime with restrictive ACL
- Send REQ from napplet without relay:read capability
- Verify the message is denied (CLOSED with denied: reason, or no events delivered)

Each test follows this pattern:
```ts

import { createRuntime } from './index.js';
import { createMockRuntimeHooks } from './test-utils.js';

describe('message dispatch', () => {
  let mock: ReturnType<typeof createMockRuntimeHooks>;
  let runtime: ReturnType<typeof createRuntime>;

  beforeEach(() => {
    mock = createMockRuntimeHooks();
    runtime = createRuntime(mock.hooks);
  });

  describe('REQ', () => {
    it('creates subscription for authenticated napplet', () => {
      // 1. Register a napplet identity (simulate completed AUTH)
      // 2. Send REQ message
      // 3. Assert subscription is tracked
      // 4. Inject matching event
      // 5. Assert sendToNapplet was called with ['EVENT', 'sub-1', event]
    });
  });

  // ... other verb tests
});
```

The exact method names for registering napplets and injecting events depend on Phase 13's runtime API. The executor must read the actual runtime interface and adapt. The key requirement is that every NIP-01 verb has at least one test, and all tests pass in Node.js without browser globals.
