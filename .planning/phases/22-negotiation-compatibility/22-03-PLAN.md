# Plan 22-03: Runtime compatibility check after AUTH

<objective>
Implement the runtime's compatibility check: after AUTH succeeds but before queued messages are dispatched, the runtime reads the napplet's manifest requires, checks them against the service registry, generates a CompatibilityReport, and either blocks loading (strict mode) or warns (permissive mode).

This satisfies NEG-01 through NEG-04 and COMPAT-03 — the core negotiation logic.

<interfaces>
The compatibility check flow after AUTH success:

1. Read the NappKeyEntry that was just registered (has pubkey, dTag, aggregateHash)
2. Look up ManifestCacheEntry by (pubkey, dTag) to get the requires list
3. For each required service name, check if it's registered in the service registry
4. If any are missing, construct a CompatibilityReport
5. Call hooks.onCompatibilityIssue(report) if report.compatible is false
6. In strict mode: send OK false, skip queued message dispatch
7. In permissive mode: proceed normally

The compatibility check function signature:
```typescript
function checkCompatibility(
  requires: string[],
  windowId: string,
  eventId: string,
): boolean  // returns true if compatible or permissive, false if strict-blocked
```

For the service registry check, the runtime needs to know what services are registered. Per Phase 18, the runtime will have a services Map. Since Phase 18 is not yet executed, we need to add a placeholder that Phase 18 will fill in. The simplest approach: add a `registeredServiceNames` Set to the runtime's internal state that tracks service names. Phase 18's registerService() will populate this. For now, initialize it empty and add a method to populate it.

Actually, since Phase 18 defines `registerService()` and the service registry, and Phase 22 depends on Phase 20 and 21 (which depend on 18/19), the service registry WILL exist by execution time. But at planning time, we need to reference it. The plan should check against whatever internal service registry exists in the runtime — if it's a Map<string, ServiceHandler>, check the keys. If Phase 18 hasn't been built yet when this plan executes (phases may build in dependency order), this plan adds its own minimal tracking.

Decision: Add a `registeredServices` Set<string> to the runtime's module-level state. Phase 18's registerService() will add to this set alongside its own data structures. This plan adds the set and the compatibility check. Phase 18's plan 02 (which adds registerService) will populate the set.
</interfaces>

  <action>
    Add a `checkCompatibility` function inside the createRuntime closure, after the `handleAuth` function (around line 257).

    ```typescript
    /**
     * Check if a napplet's declared service requirements are satisfied.
     * Called after AUTH succeeds but before queued messages are dispatched.
     *
     * Returns true if compatible (or permissive mode allows loading).
     * Returns false only in strict mode when required services are missing.
     */
    function checkCompatibility(
      requires: string[],
      windowId: string,
      eventId: string,
    ): boolean {
      if (requires.length === 0) return true;

      const available: ServiceInfo[] = Array.from(registeredServices.values());
      const registeredNames = new Set(registeredServices.keys());
      const missing = requires.filter((name) => !registeredNames.has(name));
      const compatible = missing.length === 0;

      if (!compatible) {
        const report: CompatibilityReport = { available, missing, compatible };
        hooks.onCompatibilityIssue?.(report);

        if (hooks.strictMode) {
          hooks.sendToNapplet(windowId, [
            'OK', eventId, false,
            `blocked: missing required services: ${missing.join(', ')}`,
          ]);
          return false;
        }
      }

      return true;
    }
    ```
