# Pipe Protocol — NIP-5D Section 6.4

`draft` `optional`

## Naming Rationale

The term "pipes" is used instead of "channels" to avoid collision with NIP-28 (Public Chat),
which defines relay-based public chat channels using kinds 40-44. Pipes are point-to-point
authenticated connections between napplet iframes mediated by the shell over postMessage —
a fundamentally different concept from NIP-28 relay channels.

## Wire Format

Pipe messages use the same JSON array convention as all napplet-shell communication.
The first element is the verb string.

### Napplet-to-Shell Messages

| Verb | Format | Description |
|------|--------|-------------|
| `PIPE_OPEN` | `["PIPE_OPEN", <options>]` | Request a pipe to a target napplet |
| `PIPE` | `["PIPE", <pipeId>, <payload>]` | Send data on an open pipe |
| `PIPE_CLOSE` | `["PIPE_CLOSE", <pipeId>]` | Close a pipe |
| `PIPE_BROADCAST` | `["PIPE_BROADCAST", <payload>]` | Fan out data to all pipe peers |

### Shell-to-Napplet Messages

| Verb | Format | Description |
|------|--------|-------------|
| `PIPE_ACK` | `["PIPE_ACK", <info>]` | Pipe opened — contains pipeId and peer identity |
| `PIPE` | `["PIPE", <pipeId>, <payload>]` | Incoming data from pipe peer |
| `PIPE_CLOSE` | `["PIPE_CLOSE", <pipeId>, <reason>]` | Pipe closed by peer or shell |

### PIPE_OPEN Options

```json
["PIPE_OPEN", {
  "targetType": "synth-plugin",
  "targetKey": "abc123def456...",
  "name": "audio-data"
}]
```

- `targetType` (string, REQUIRED): dTag of the target napplet type
- `targetKey` (string, optional): Session pubkey of a specific instance when multiple
  instances of the same type are loaded. If omitted, the shell selects an instance.
- `name` (string, optional): Named pipe for multiplexing multiple logical pipes to the
  same target

### PIPE_ACK Info

```json
["PIPE_ACK", {
  "pipeId": "p-7f3a",
  "peer": {"pubkey": "def456...", "dTag": "synth-plugin"}
}]
```

- `pipeId` (string, REQUIRED): Shell-assigned opaque identifier for this pipe
- `peer` (object, REQUIRED): Identity of the connected peer
  - `peer.pubkey` (string): Peer's session public key
  - `peer.dTag` (string): Peer's napplet type

### PIPE Data

```json
["PIPE", "p-7f3a", {"bpm": 120, "beat": 4}]
```

- Position 1: pipeId (string) — the shell routes by this identifier
- Position 2: payload (any JSON-serializable value) — opaque to the shell

### PIPE_CLOSE

```json
["PIPE_CLOSE", "p-7f3a"]
["PIPE_CLOSE", "p-7f3a", "peer-disconnected"]
```

- Position 1: pipeId (string)
- Position 2 (optional): reason string. Present when the shell or peer initiated the close.

### PIPE_BROADCAST

```json
["PIPE_BROADCAST", {"bpm": 120}]
```

- Position 1: payload (any JSON-serializable value)
- Shell delivers as `["PIPE", "<pipeId>", <payload>]` to each open pipe peer
- Sender is excluded from delivery

## Lifecycle

### Pipe Establishment (Shell-Mediated)

The shell auto-connects pipes. The target napplet does not explicitly accept — the shell
is authoritative about pipe creation after verifying both endpoints are authenticated.

```
Napplet A                    Shell                      Napplet B
    |                          |                            |
    |--["PIPE_OPEN", opts]---->|                            |
    |                          |  (1) verify A AUTH'd       |
    |                          |  (2) check ACL             |
    |                          |  (3) resolve target by     |
    |                          |      targetType + targetKey|
    |                          |  (4) assign pipeId         |
    |                          |                            |
    |<-["PIPE_ACK", info]------|--["PIPE_ACK", info]------->|
    |                          |                            |
    |--["PIPE", id, data]----->|--["PIPE", id, data]------->|
    |<-["PIPE", id, data]------|<-["PIPE", id, data]--------|
    |                          |                            |
    |--["PIPE_CLOSE", id]----->|                            |
    |                          |--["PIPE_CLOSE", id, ""]--->|
    |<-["PIPE_CLOSE", id, ""]--|                            |
```

Steps:
1. Napplet A sends PIPE_OPEN with target dTag (and optional targetKey)
2. Shell verifies A has completed AUTH handshake
3. Shell checks ACL for pipe capability (if shell implements ACL)
4. Shell resolves the target napplet by dTag, assigns an opaque pipeId
5. Shell sends PIPE_ACK to BOTH endpoints simultaneously
6. Either side may now send PIPE messages
7. Either side may send PIPE_CLOSE to tear down the pipe

### Broadcast Flow

```
Napplet A                    Shell                 Napplet B    Napplet C
    |                          |                       |            |
    |--["PIPE_BROADCAST", d]-->|                       |            |
    |                          |--["PIPE","pB",d]----->|            |
    |                          |--["PIPE","pC",d]----->|----------->|
    |                          |                       |            |
```

Shell iterates all open pipes where A is an endpoint and delivers the payload to each
peer. Sender A does NOT receive its own broadcast.

### Error Handling

| Error | Trigger | Shell Response |
|-------|---------|----------------|
| Not authenticated | PIPE_OPEN before AUTH completes | `["PIPE_CLOSE", null, "auth-required"]` |
| Target not found | No napplet loaded with targetType | `["PIPE_CLOSE", null, "target-not-found"]` |
| ACL denied | Pipe capability not granted | `["PIPE_CLOSE", null, "acl-denied"]` |
| Peer disconnected | Target window destroyed | `["PIPE_CLOSE", "<pipeId>", "peer-disconnected"]` |
| Invalid pipe | PIPE sent with unknown pipeId | Silently dropped |

For errors during PIPE_OPEN (before pipeId assignment), the shell sends PIPE_CLOSE with
`null` as the pipeId. The initiator correlates by the error context — only one PIPE_OPEN
is outstanding per call.

## Capability Discovery

Pipes are a MAY capability. Napplets MUST check for pipe support before use:

```json
["REQ", "cap", {"kinds": [29010]}]

["EVENT", "cap", {
  "kind": 29010,
  "tags": [["s", "pipes"], ["v", "1.0.0"]],
  ...
}]
```

Or programmatically:

```js
if (window.napplet.services.has('pipes')) {
  const pipe = await window.napplet.pipes.open('synth-plugin');
}
```

## API Surface

Shells providing the pipes capability MUST expose `window.napplet.pipes`:

| Method | Returns | Description |
|--------|---------|-------------|
| `open(targetType, options?)` | `Promise<PipeHandle>` | Open a pipe to a target napplet type |
| `onOpen(callback)` | `{close()}` | Listen for incoming pipe connections |
| `broadcast(payload)` | `void` | Fan out to all pipe peers |

`options` object: `{ targetKey?: string, name?: string }`

A `PipeHandle` provides:

| Property/Method | Type | Description |
|-----------------|------|-------------|
| `id` | `string` | Shell-assigned pipe identifier |
| `peer` | `{pubkey, dTag}` | Connected peer identity |
| `send(payload)` | `void` | Send data to the peer |
| `on(callback)` | `{close()}` | Listen for incoming data |
| `close()` | `void` | Close the pipe |
| `onClose(callback)` | `void` | Listen for close notification |

## Security Considerations

- Pipes require completed AUTH handshake — unauthenticated napplets MUST NOT open pipes
- All pipe messages flow through the shell — no direct iframe-to-iframe communication
- Shell SHOULD enforce ACL for pipe creation (capability string: `pipe:connect`)
- Pipe payloads are opaque to the shell — the shell routes but does not inspect content
- Shell identifies senders by MessageEvent.source (unforgeable Window reference)
