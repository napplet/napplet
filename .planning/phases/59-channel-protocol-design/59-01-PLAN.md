  ```
  ## Extension Discovery

  Extension discovery is MUST. All shells MUST implement `shell.supports()`.

  Napplets discover shell capabilities by querying NUB proposal support:

  - `shell.supports("NUB-RELAY")` -- boolean, interface present
  - `shell.supports("NUB-RELAY", "NUB-02")` -- boolean, interface + protocol

  How shells resolve `supports()` internally is an implementation detail. This
  NIP does not prescribe a wire protocol for discovery.

  Napplets MUST discover capabilities before using them. Napplets MUST gracefully
  degrade when a capability is absent.

  ### Requirement Declaration

  Napplet manifests ([NIP-5A](5A.md) kind 35128) MAY include
  `["requires", "<nub-id>"]` tags declaring which capabilities the napplet needs.
  The shell checks these during the post-AUTH compatibility check.
  ```

**ADD new section before Security Considerations:**
```
## NUB Extension Framework

Capabilities beyond the core protocol (authentication, transport, security) are
defined as NUB (Napplet Unified Blueprint) proposals. The NUB system has two
tracks:

**NUB-WORD** (interfaces): One canonical spec per name. Defines shell-provided
API contracts on `window.napplet.*` namespaces. Examples: NUB-RELAY, NUB-STORAGE,
NUB-SIGNER, NUB-NOSTRDB, NUB-IPC, NUB-PIPES.

**NUB-NN** (message protocols): Numbered proposals. Multiple competing specs
allowed per domain. Defines event semantics that napplets agree on with each
other. Napplets negotiate via extension discovery.

NUB proposals are maintained at [github.com/napplets](https://github.com/napplets).
Each proposal is a self-contained markdown document following the NUB template.

Event kinds used by NUB interfaces (29001-29010 in the reference implementation)
are defined in their respective NUB proposals, not in this NIP.
```
Per D-03, D-05, D-06.
