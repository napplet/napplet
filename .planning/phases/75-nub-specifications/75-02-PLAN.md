Create specs/nubs/NUB-STORAGE.md defining scoped key-value storage messages within the JSON envelope.

Match NIP-5D's prescriptive style. The current implementation uses topic-string IPC events (shell:state-get, shell:state-set, etc.) with NIP-01 tag arrays. This NUB replaces that with explicit JSON envelope messages.

The spec MUST contain these sections in this order:

**1. Header**
```
NUB-STORAGE
============

Scoped Storage
--------------

`draft` `optional`

This NUB defines scoped key-value storage messages for the napplet shell protocol. It enables napplets to persistently store and retrieve string values through the shell using the [NIP-5D](../NIP-5D.md) JSON envelope wire format. Storage is scoped per napplet identity -- napplets cannot access each other's data.
```

**2. Terminology table**
| Term | Definition |
|------|------------|
| Request ID | Client-generated UUID correlating a request with its response |
| Storage scope | The shell-enforced namespace for a napplet's storage, derived from its identity (dTag + aggregateHash) |
| Quota | Maximum total storage size per napplet, enforced by the shell |

**3. Capability section**
```
## Capability

Manifest tag: `["requires", "storage"]`

Runtime check: `window.napplet.shell.supports('storage')`
```

**4. Messages section** with the exact message catalog:

**Napplet-to-Shell Messages (requests):**

`storage.get` -- Retrieve a value
```json
{
  "type": "storage.get",
  "id": "<uuid>",
  "key": "<string>"
}
```
The shell looks up the key within the napplet's storage scope. The shell MUST respond with `storage.get.result`.

`storage.set` -- Store a value
```json
{
  "type": "storage.set",
  "id": "<uuid>",
  "key": "<string>",
  "value": "<string>"
}
```
The shell stores the key-value pair within the napplet's storage scope. The shell MUST respond with `storage.set.result` or `storage.error`. If the write would exceed the napplet's quota, the shell MUST respond with `storage.error`.

`storage.delete` -- Remove a key
```json
{
  "type": "storage.delete",
  "id": "<uuid>",
  "key": "<string>"
}
```
The shell removes the key from the napplet's storage scope. Deleting a nonexistent key is silently successful. The shell MUST respond with `storage.delete.result`.

`storage.keys` -- List all keys
```json
{
  "type": "storage.keys",
  "id": "<uuid>"
}
```
The shell returns all keys in the napplet's storage scope. The shell MUST respond with `storage.keys.result`.

**Shell-to-Napplet Messages (responses):**

`storage.get.result` -- Value retrieval response
```json
{
  "type": "storage.get.result",
  "id": "<uuid>",
  "value": "<string>" | null
}
```
Returns the stored value, or `null` if the key does not exist.

`storage.set.result` -- Write acknowledgement
```json
{
  "type": "storage.set.result",
  "id": "<uuid>"
}
```
Confirms the key-value pair was stored.

`storage.delete.result` -- Deletion acknowledgement
```json
{
  "type": "storage.delete.result",
  "id": "<uuid>"
}
```
Confirms the key was removed (or did not exist).

`storage.keys.result` -- Key listing response
```json
{
  "type": "storage.keys.result",
  "id": "<uuid>",
  "keys": ["<string>", ...]
}
```
Returns all keys in the napplet's storage scope. Empty array if no keys exist.

`storage.error` -- Error response (applies to any request)
```json
{
  "type": "storage.error",
  "id": "<uuid>",
  "error": "<human-readable reason>"
}
```
Sent when a storage request fails (quota exceeded, ACL denied, internal error). Known error reasons: `"quota exceeded"`, `"storage:read capability denied"`, `"storage:write capability denied"`.

**5. Storage Scoping section**
- Storage is scoped by the napplet's identity tuple: `(dTag, aggregateHash)` from the NIP-5A manifest
- Different napplet types have isolated storage
- Different versions of the same napplet type (different aggregate hashes) have isolated storage
- The shell derives the scope from the napplet's identity at iframe creation time (see NIP-5D Identity section)
- Key names are arbitrary strings; the shell MUST NOT interpret key names beyond scoping

**6. Quotas section**
- Shells MUST enforce a per-napplet storage quota
- The recommended default quota is 512 KB per napplet
- Quota is measured as the sum of all key lengths plus all value lengths within the scope
- Quota enforcement is checked on `storage.set`; the shell MUST reject the write with `storage.error` if it would exceed the quota
- Shells MAY expose quota configuration to users

**7. Shell Behavior section**
- Shell MUST enforce ACL: `state:read` for get and keys operations, `state:write` for set and delete operations
- Shell MUST scope all operations to the requesting napplet's identity
- Shell MUST persist storage across page reloads (typically via localStorage, IndexedDB, or equivalent)
- Shell MUST respond to every request with exactly one response (result or error)
- Shell SHOULD respond within a reasonable timeframe; napplets SHOULD implement a timeout (recommended: 5 seconds)

**8. Security Considerations section**
- Scope isolation prevents cross-napplet data access
- Aggregate hash scoping means a tampered napplet build cannot read the legitimate version's data
- Quota enforcement prevents storage exhaustion attacks
- All values are strings; the shell does not interpret content (binary data should be base64-encoded by the napplet)

**9. References section**
- [NIP-5D](../NIP-5D.md) -- Wire format and transport
- [NIP-5A](https://github.com/nostr-protocol/nips/blob/master/5A.md) -- Napplet manifest and aggregate hash

Target: ~110-140 lines, readable in under 5 minutes.

Create specs/nubs/NUB-IFC.md defining inter-frame communication messages within the JSON envelope.

This is the most conceptually complex NUB. It merges the former IPC (fire-and-forget peer events) and PIPES (named channel pubsub) concepts into a single spec with two modes:
- **Dispatch mode**: fire-and-forget messages, each goes through ACL individually
- **Channel mode**: opened once with ACL check at open, subsequent messages flow without per-message ACL

Match NIP-5D's prescriptive style.

The spec MUST contain these sections in this order:

**1. Header**
```
NUB-IFC
=======

Inter-Frame Communication
-------------------------

`draft` `optional`

This NUB defines inter-frame communication messages for the napplet shell protocol. It enables napplets to exchange messages with each other through the shell using the [NIP-5D](../NIP-5D.md) JSON envelope wire format. Two modes are provided: dispatch (per-message routing with individual ACL checks) and channel (persistent topic subscriptions with ACL checked at open time).
```

**2. Terminology table**
| Term | Definition |
|------|------------|
| Topic | A string identifier for message routing (e.g., `profile:open`, `chat:message`) |
| Dispatch | Fire-and-forget message delivery; each message is individually ACL-checked and routed |
| Channel | Persistent topic subscription; ACL is checked once at open time, subsequent messages flow freely |
| Channel ID | Client-generated UUID identifying an open channel subscription |

**3. Capability section**
```
## Capability

Manifest tag: `["requires", "ifc"]`

Runtime check: `window.napplet.shell.supports('ifc')`
```

**4. Dispatch Mode section** with message catalog:

Dispatch mode is the simple case: a napplet sends a message to a topic, and the shell routes it to all napplets subscribed to that topic. Each message is individually checked against ACL.

**Napplet-to-Shell:**

`ifc.dispatch` -- Send a message to a topic
```json
{
  "type": "ifc.dispatch",
  "topic": "<string>",
  "payload": <any JSON value>
}
```
The shell routes the message to all napplets that have an active subscription (dispatch listener or open channel) on the given topic. The sender does NOT receive its own message. The shell ACL-checks each dispatch individually. No response is sent (fire-and-forget).

`ifc.dispatch.listen` -- Subscribe to a topic (dispatch mode)
```json
{
  "type": "ifc.dispatch.listen",
  "topic": "<string>"
}
```
The napplet registers interest in messages on the given topic. The shell delivers matching `ifc.dispatch` messages from other napplets as `ifc.message` events. No response is sent. Listening to the same topic twice is idempotent.

`ifc.dispatch.unlisten` -- Unsubscribe from a topic (dispatch mode)
```json
{
  "type": "ifc.dispatch.unlisten",
  "topic": "<string>"
}
```
The napplet removes interest in the given topic. No response is sent. Unlistening from a topic not subscribed to is silently ignored.

**Shell-to-Napplet (delivery for both modes):**

`ifc.message` -- Deliver a routed message
```json
{
  "type": "ifc.message",
  "topic": "<string>",
  "payload": <any JSON value>,
  "source": "<napplet dTag>"
}
```
Delivered to subscribers (dispatch listeners or open channels) when a message arrives on a matching topic. The `source` field identifies the sending napplet by its dTag. The `payload` is the value from the original `ifc.dispatch` or `ifc.channel.send`.

**5. Channel Mode section** with message catalog:

Channel mode provides persistent topic subscriptions with ACL checked once at open time. This is suitable for high-frequency message streams where per-message ACL overhead is undesirable.

**Napplet-to-Shell:**

`ifc.channel.open` -- Open a channel on a topic
```json
{
  "type": "ifc.channel.open",
  "id": "<uuid>",
  "topic": "<string>"
}
```
The shell ACL-checks the napplet for IFC access and, if permitted, opens a persistent subscription on the given topic. The shell MUST respond with `ifc.channel.opened` or `ifc.channel.error`. Messages on the topic are delivered as `ifc.message` for the lifetime of the channel.

`ifc.channel.send` -- Send a message on an open channel
```json
{
  "type": "ifc.channel.send",
  "id": "<uuid>",
  "payload": <any JSON value>
}
```
Send a message through an open channel. The `id` is the channel ID from `ifc.channel.open`. The shell routes to all subscribers on the channel's topic. No per-message ACL check. No response is sent. If the channel ID is unknown, the message is silently dropped.

`ifc.channel.close` -- Close a channel
```json
{
  "type": "ifc.channel.close",
  "id": "<uuid>"
}
```
Close the channel and stop receiving messages. No response is sent. Closing an unknown channel ID is silently ignored.

**Shell-to-Napplet:**

`ifc.channel.opened` -- Channel open acknowledgement
```json
{
  "type": "ifc.channel.opened",
  "id": "<uuid>",
  "topic": "<string>"
}
```
Confirms the channel is open and messages will be delivered.

`ifc.channel.error` -- Channel open failure
```json
{
  "type": "ifc.channel.error",
  "id": "<uuid>",
  "error": "<human-readable reason>"
}
```
The channel could not be opened (ACL denied, invalid topic, shell limit reached).

**6. Topic Conventions section**
- Topics are arbitrary strings; the shell does not interpret them beyond routing
- Recommended format: `namespace:action` (e.g., `profile:open`, `chat:message`, `stream:channel-switch`)
- Topics prefixed with `shell:` are reserved for shell-internal use and MUST NOT be used by napplets
- Topics prefixed with `napplet:` are reserved for shell-to-napplet notifications
- Napplets SHOULD use descriptive, namespaced topics to avoid collisions

**7. Shell Behavior section**
- Shell MUST route dispatch messages only to napplets with active listeners or open channels on the matching topic
- Shell MUST NOT deliver a message back to its sender
- In dispatch mode, shell checks ACL on each `ifc.dispatch` message
- In channel mode, shell checks ACL once on `ifc.channel.open`; subsequent `ifc.channel.send` messages bypass ACL
- Shell MAY impose limits on: number of dispatch listeners per napplet, number of open channels per napplet, message rate per topic
- Shell MUST clean up all listeners and channels when a napplet iframe is destroyed

**8. Dispatch vs Channel Guidance section**
Provide a brief table:

| Aspect | Dispatch | Channel |
|--------|----------|---------|
| ACL check | Per message | At open time |
| Overhead | Higher (ACL per message) | Lower (ACL once) |
| Use case | Infrequent commands, one-off notifications | Real-time streams, chat, collaborative state |
| Subscription | `ifc.dispatch.listen` | `ifc.channel.open` |
| Teardown | `ifc.dispatch.unlisten` | `ifc.channel.close` |

**9. Security Considerations section**
- ACL enforcement prevents unauthorized inter-frame communication
- Source identification via dTag allows receivers to trust or filter senders
- Channel mode trades per-message ACL for performance; shells SHOULD only allow channel mode for trusted napplets
- Topic isolation: shell routes by exact topic match, not prefix match
- No message persistence: IFC messages are ephemeral and not stored

**10. References section**
- [NIP-5D](../NIP-5D.md) -- Wire format and transport
