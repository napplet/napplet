Match NIP-5D's prescriptive style: markdown with tables, indented code blocks for wire examples, MUST/SHOULD/MAY language.

The spec MUST contain these sections in this order:

**1. Header**
```
NUB-RELAY
=========

Relay Proxy
-----------

`draft` `optional`

This NUB defines the relay proxy messages for the napplet shell protocol. It enables napplets to subscribe to events, publish events, and perform count queries through the shell's relay pool using the [NIP-5D](../NIP-5D.md) JSON envelope wire format.
```

**2. Terminology table**
| Term | Definition |
|------|------------|
| Subscription ID | Client-generated UUID identifying a live subscription |
| Filter | NIP-01 filter object (kinds, authors, ids, tag filters, since, until, limit) |
| Event template | Unsigned event object (kind, content, tags, created_at) without id, pubkey, or sig |

**3. Capability section**
```
## Capability

Manifest tag: `["requires", "relay"]`

Runtime check: `window.napplet.shell.supports('relay')`
```

**4. Messages section** with subsections for each message type. Use this exact message catalog:

**Napplet-to-Shell Messages:**

`relay.subscribe` -- Open a subscription
```json
{
  "type": "relay.subscribe",
  "subscription": "<uuid>",
  "filters": [<filter>, ...]
}
```
The shell opens the subscription against its relay pool and/or local cache. The `filters` array contains one or more NIP-01 filter objects. The shell MUST respond with zero or more `relay.event` messages followed by exactly one `relay.eose`.

`relay.close` -- Close a subscription
```json
{
  "type": "relay.close",
  "subscription": "<uuid>"
}
```
The shell closes the named subscription. No response is sent. Closing an unknown subscription ID is silently ignored.

`relay.publish` -- Publish an event
```json
{
  "type": "relay.publish",
  "event": {
    "kind": <number>,
    "content": "<string>",
    "tags": [["tag", "value"], ...],
    "created_at": <unix_seconds>
  }
}
```
The event is an unsigned template. The shell stamps identity (id, pubkey, sig) and broadcasts to connected relays. The shell MUST respond with `relay.ok` indicating success or failure. Destructive kinds (0, 3, 5, 10002) MAY trigger a consent prompt before signing.

`relay.count` -- Request event count
```json
{
  "type": "relay.count",
  "subscription": "<uuid>",
  "filters": [<filter>, ...]
}
```
NIP-45 count request. The shell MUST respond with `relay.count.result`.

**Shell-to-Napplet Messages:**

`relay.event` -- Deliver a matching event
```json
{
  "type": "relay.event",
  "subscription": "<uuid>",
  "event": {<signed NostrEvent>}
}
```
Delivered for each event matching an active subscription's filters. The event is a fully signed NIP-01 event (id, pubkey, kind, content, tags, created_at, sig).

`relay.eose` -- End of stored events
```json
{
  "type": "relay.eose",
  "subscription": "<uuid>"
}
```
Signals that all stored events matching the subscription have been delivered. Live events may follow.

`relay.ok` -- Publish acknowledgement
```json
{
  "type": "relay.ok",
  "id": "<event_id>",
  "ok": true|false,
  "reason": "<string>"
}
```
Sent after `relay.publish`. `id` is the event ID assigned by the shell after signing. `ok` is true on success, false on rejection. `reason` is a human-readable string (empty on success).

`relay.closed` -- Subscription terminated by shell
```json
{
  "type": "relay.closed",
  "subscription": "<uuid>",
  "reason": "<string>"
}
```
The shell terminated the subscription (rate limit, ACL revocation, etc.). The napplet SHOULD NOT send further messages on this subscription ID.

`relay.notice` -- Informational message
```json
{
  "type": "relay.notice",
  "message": "<string>"
}
```
Non-fatal informational message from the shell. Not tied to a specific subscription.

`relay.count.result` -- Count response
```json
{
  "type": "relay.count.result",
  "subscription": "<uuid>",
  "count": <number>
}
```
Response to `relay.count`. The count MAY be approximate.

**5. Filter Format section**
Document that filters follow NIP-01 semantics:
```json
{
  "ids": ["<hex event id>", ...],
  "authors": ["<hex pubkey>", ...],
  "kinds": [<integer>, ...],
  "#t": ["<tag value>", ...],
  "since": <unix_seconds>,
  "until": <unix_seconds>,
  "limit": <integer>
}
```
All fields are optional. Tag filters use the `#<single-letter>` key format per NIP-01. Multiple filters in an array are OR'd; fields within a filter are AND'd.

**6. Shell Behavior section**
- Shell MUST validate filters before opening subscriptions
- Shell MUST enforce ACL: `relay:read` for subscribe/count, `relay:write` for publish
- Shell routes subscriptions to its relay pool; implementation of pool management is out of scope
- Shell MAY impose per-napplet subscription limits
- Shell MAY reject publish for events failing validation (malformed tags, invalid kind)

**7. Security Considerations section**
- Napplets never see signing keys; shell stamps identity on publish
- ACL enforcement prevents unauthorized relay access
- Subscription IDs are napplet-scoped; one napplet cannot close another's subscriptions
- Rate limiting SHOULD be applied per napplet

**8. References section**
- [NIP-5D](../NIP-5D.md) -- Wire format and transport
- [NIP-01](https://github.com/nostr-protocol/nips/blob/master/01.md) -- Filter semantics
- [NIP-45](https://github.com/nostr-protocol/nips/blob/master/45.md) -- Event count

The spec MUST contain these sections in this order:

**1. Header**
```
NUB-SIGNER
==========

Signing Delegation
------------------

`draft` `optional`

This NUB defines signing delegation messages for the napplet shell protocol. It enables napplets to request cryptographic operations (event signing, public key retrieval, NIP-04/NIP-44 encryption and decryption) through the shell using the [NIP-5D](../NIP-5D.md) JSON envelope wire format. The shell delegates to its backing signer (NIP-07 extension, NIP-46 remote signer, or other implementation).
```

**2. Terminology table**
| Term | Definition |
|------|------------|
| Request ID | Client-generated UUID correlating a request with its response |
| Backing signer | The shell's signing implementation (NIP-07 extension, NIP-46 remote, hardware key, etc.) |
| Event template | Unsigned event object (kind, content, tags, created_at) without id, pubkey, or sig |

**3. Capability section**
```
## Capability

Manifest tag: `["requires", "signer"]`

Runtime check: `window.napplet.shell.supports('signer')`
```

**4. Messages section** with the exact message catalog:

**Napplet-to-Shell Messages (requests):**

`signer.get-public-key` -- Request the user's public key
```json
{
  "type": "signer.get-public-key",
  "id": "<uuid>"
}
```
The shell MUST respond with `signer.get-public-key.result`.

`signer.sign` -- Request event signing
```json
{
  "type": "signer.sign",
  "id": "<uuid>",
  "event": {
    "kind": <number>,
    "content": "<string>",
    "tags": [["tag", "value"], ...],
    "created_at": <unix_seconds>
  }
}
```
The event is an unsigned template. The shell signs via its backing signer and returns the fully signed event. Destructive kinds (0, 3, 5, 10002) MAY trigger a consent prompt. The shell MUST respond with `signer.sign.result` or `signer.error`.

`signer.nip04.encrypt` -- NIP-04 encryption
```json
{
  "type": "signer.nip04.encrypt",
  "id": "<uuid>",
  "pubkey": "<hex recipient pubkey>",
  "plaintext": "<string>"
}
```

`signer.nip04.decrypt` -- NIP-04 decryption
```json
{
  "type": "signer.nip04.decrypt",
  "id": "<uuid>",
  "pubkey": "<hex sender pubkey>",
  "ciphertext": "<string>"
}
```

`signer.nip44.encrypt` -- NIP-44 encryption
```json
{
  "type": "signer.nip44.encrypt",
  "id": "<uuid>",
  "pubkey": "<hex recipient pubkey>",
  "plaintext": "<string>"
}
```

`signer.nip44.decrypt` -- NIP-44 decryption
```json
{
  "type": "signer.nip44.decrypt",
  "id": "<uuid>",
  "pubkey": "<hex sender pubkey>",
  "ciphertext": "<string>"
}
```

**Shell-to-Napplet Messages (responses):**

`signer.get-public-key.result` -- Public key response
```json
{
  "type": "signer.get-public-key.result",
  "id": "<uuid>",
  "pubkey": "<hex public key>"
}
```

`signer.sign.result` -- Signed event response
```json
{
  "type": "signer.sign.result",
  "id": "<uuid>",
  "event": {<signed NostrEvent>}
}
```

`signer.nip04.encrypt.result` / `signer.nip04.decrypt.result` / `signer.nip44.encrypt.result` / `signer.nip44.decrypt.result` -- Crypto operation results
```json
{
  "type": "signer.<operation>.result",
  "id": "<uuid>",
  "result": "<string>"
}
```
Where `<operation>` is one of `nip04.encrypt`, `nip04.decrypt`, `nip44.encrypt`, `nip44.decrypt`. The `result` field contains the ciphertext or plaintext string.

`signer.error` -- Error response (applies to any request)
```json
{
  "type": "signer.error",
  "id": "<uuid>",
  "error": "<human-readable reason>"
}
```
Sent when a signer request fails (backing signer unavailable, user denied consent, invalid parameters). The `id` correlates to the original request.

**5. Request/Response Pattern section**
- Every request includes a client-generated `id` (UUID)
- The shell MUST respond with exactly one response per request: either `*.result` or `signer.error`
- Responses MUST include the matching `id`
- If the backing signer is unavailable, the shell MUST respond with `signer.error`, not silently drop
- Napplets SHOULD implement a timeout (recommended: 30 seconds) for signer requests

**6. Shell Behavior section**
- Shell MUST enforce ACL: `sign:event` for sign requests, `sign:nip04` for NIP-04, `sign:nip44` for NIP-44
- Public key retrieval does not require a specific capability
- Shell delegates to its backing signer; the choice of NIP-07, NIP-46, or other is an implementation detail
- Shell MUST NOT expose the signing private key to the napplet
- Consent prompts for destructive kinds are at shell discretion

**7. Relationship to window.nostr section**
Note that shells also provide a NIP-07 `window.nostr` implementation to napplet iframes (per NIP-5D). The `window.nostr` API and these signer messages are two interfaces to the same backing signer. Shells MAY implement `window.nostr` by internally dispatching signer.* envelope messages, or MAY implement both paths independently. Napplets SHOULD prefer `window.nostr` for NIP-07 compatibility; the envelope messages exist for programmatic access without the DOM global.

**8. Security Considerations section**
- Napplets never access signing keys; all operations are delegated
- ACL enforcement prevents unauthorized signing
- Correlation IDs prevent response spoofing (napplet ignores responses with unknown IDs)
- Consent prompts for destructive kinds protect against malicious napplets modifying user metadata or contact lists

**9. References section**
- [NIP-5D](../NIP-5D.md) -- Wire format and transport
- [NIP-07](https://github.com/nostr-protocol/nips/blob/master/07.md) -- window.nostr signer
- [NIP-04](https://github.com/nostr-protocol/nips/blob/master/04.md) -- Encrypted direct messages (deprecated, included for compatibility)
- [NIP-44](https://github.com/nostr-protocol/nips/blob/master/44.md) -- Versioned encryption
- [NIP-46](https://github.com/nostr-protocol/nips/blob/master/46.md) -- Remote signing
