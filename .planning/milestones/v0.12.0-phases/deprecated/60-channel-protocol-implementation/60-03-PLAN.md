Pipe wire format (renamed from CH_* to PIPE_*):
  PIPE_OPEN: ["PIPE_OPEN", pipeId, {target: dTag, name: string}]
  PIPE_ACK: ["PIPE_ACK", pipeId, {peer: pubkey, peerDTag: dTag}]
  PIPE: ["PIPE", pipeId, payload]
  PIPE_CLOSE: ["PIPE_CLOSE", pipeId]
  PIPE_CLOSED: ["PIPE_CLOSED", pipeId, reason]
  PIPE_ERROR: ["PIPE_ERROR", pipeId, errorMessage]
  PIPE_BROADCAST: ["PIPE_BROADCAST", groupOrStar, payload]

Proposed shim API namespace: window.napplet.pipes
  open(target, name?): Promise<PipeHandle>
  onOpen(callback): Subscription
  broadcast(payload, group?): void

PipeHandle:
  id: string (shell-assigned)
  peer: {pubkey, dTag}
  name: string
  send(payload): void
  on(callback): Subscription
  close(): void
  onClose(callback): void

Key design decisions from context:
- Shell-mediated: no direct iframe-to-iframe
- Auth-on-open: reuse existing AUTH session identity, no per-message signing
- Minimal post-auth envelope: ["PIPE", id, payload] ~30 bytes vs ~500 bytes for kind 29003
- Shell-assigned pipe IDs (UUID or incrementing integer)
- Targeting by dTag (napplet type identifier)
- Broadcast to all open pipes (group="*") or named group
- Pre-open message queuing (capped at 50) for startup race conditions
- MessagePort upgrade as MAY future optimization

Create specs/nubs/NUB-PIPES.md as the most detailed NUB-WORD spec. This is a new unimplemented protocol so it gets full treatment per D-01 and D-02. Content:

**Header:**
```
NUB-PIPES
=========

Authenticated Point-to-Point Connections
-----------------------------------------

`draft` `unimplemented`

**NUB ID:** NUB-PIPES
**Namespace:** `window.napplet.pipes`
**Discovery:** `shell.supports("NUB-PIPES")`
```

Note the extra `unimplemented` badge per D-02.

**Description:** Two paragraphs. First: NUB-PIPES provides authenticated, persistent, bidirectional connections between napplets. Unlike NUB-IPC (topic-based pub/sub with per-message signing), pipes authenticate once on open and then exchange messages with minimal overhead. The shell mediates all pipe operations -- napplets cannot connect directly due to iframe sandbox restrictions.

Second: Pipes are designed for high-frequency communication where per-message NIP-01 event signing (as in NUB-IPC) creates unacceptable overhead. A kind 29003 IPC message costs ~500 bytes of envelope plus Schnorr signature computation. A pipe message costs ~30 bytes. Use NUB-IPC for infrequent coordination; use NUB-PIPES for data streams, real-time sync, and command channels.

**API Surface:**
```typescript
interface NappletPipes {
  open(target: string, name?: string): Promise<PipeHandle>;
  onOpen(callback: (pipe: PipeHandle) => void): Subscription;
  broadcast(payload: unknown, group?: string): void;
}

interface PipeHandle {
  readonly id: string;
  readonly peer: { pubkey: string; dTag: string };
  readonly name: string;
  send(payload: unknown): void;
  on(callback: (payload: unknown) => void): Subscription;
  close(): void;
  onClose(callback: (reason: string) => void): void;
}

interface Subscription {
  close(): void;
}
```

Method descriptions:
- `open(target, name?)` opens a pipe to a napplet identified by its dTag. `name` is an optional channel name for multiplexing (e.g., "midi", "transport", "state"). Returns a Promise that resolves when the target accepts the pipe.
- `onOpen(callback)` listens for incoming pipe open requests from other napplets. The callback receives a PipeHandle that is already connected.
- `broadcast(payload, group?)` sends a message to all open pipes. If `group` is specified, only pipes with matching name receive it. `"*"` broadcasts to all.
- `PipeHandle.send(payload)` sends a JSON-serializable value to the peer.
- `PipeHandle.on(callback)` receives messages from the peer.
- `PipeHandle.close()` gracefully closes the pipe. Both sides are notified.
- `PipeHandle.onClose(callback)` fires when the peer closes or disconnects.

**Wire Format:** Pipe messages are NOT NIP-01 events. They use a minimal JSON array envelope after auth-on-open:

### Control Messages

```
["PIPE_OPEN", <pipe_id>, {"target": "<dTag>", "name": "<name>"}]
["PIPE_ACK", <pipe_id>, {"peer": "<pubkey>", "peerDTag": "<dTag>"}]
["PIPE_CLOSE", <pipe_id>]
["PIPE_CLOSED", <pipe_id>, "<reason>"]
["PIPE_ERROR", <pipe_id>, "<error_message>"]
```

### Data Messages

```
["PIPE", <pipe_id>, <payload>]
```

`payload` is any JSON-serializable value. The shell does not interpret payload content.

### Broadcast Messages

```
["PIPE_BROADCAST", "<group_or_*>", <payload>]
```

The shell delivers broadcast as `["PIPE", <pipe_id>, <payload>]` to each recipient. The sender is excluded (matching NUB-IPC sender exclusion precedent).

**Pipe Lifecycle:**

```
Napplet A                    Shell                      Napplet B
    |                          |                            |
    |--PIPE_OPEN, id, opts---->|                            |
    |                          |  (verify AUTH,             |
    |                          |   check ACL, find target)  |
    |                          |                            |
    |                          |--PIPE_OPEN, id, info------>|
    |                          |                            |
    |                          |<--PIPE_ACK, id, info-------|
    |<--PIPE_ACK, id, info-----|                            |
    |                          |                            |
    |--PIPE, id, payload------>|--PIPE, id, payload-------->|
    |<--PIPE, id, payload------|<--PIPE, id, payload--------|
    |                          |                            |
    |--PIPE_CLOSE, id--------->|--PIPE_CLOSED, id, ""------>|
    |<--PIPE_CLOSED, id, ""----|                            |
```

State machine: `CLOSED -> OPENING -> OPEN -> CLOSING -> CLOSED`. The shell validates that both endpoints have completed the napplet AUTH handshake (REGISTER/IDENTITY/AUTH) before processing PIPE_OPEN.

**Shell Behavior:**
- The shell MUST assign pipe IDs. Napplets propose IDs in PIPE_OPEN; the shell MAY reassign.
- The shell MUST verify that both endpoints have completed AUTH before establishing a pipe.
- The shell MUST forward PIPE messages between connected peers without modification.
- The shell MUST notify both sides on disconnect via PIPE_CLOSED.
- The shell MUST send PIPE_ERROR for invalid operations (unknown pipe ID, target not found, ACL denied).
- The shell MAY enforce ACL checks on a `pipe:connect` capability.
- The shell MAY restrict pipe targets to specific dTags via ACL.
- The shell SHOULD queue messages sent before the target completes AUTH (max 50 messages, matching post-AUTH queue precedent).
- The shell SHOULD clean up pipe state when a napplet iframe is destroyed.

**Future Extensions:**
- **MessagePort upgrade (MAY):** After PIPE_ACK, the shell MAY transfer a MessagePort to each endpoint for direct communication, bypassing the shell's message listener. This is a performance optimization for high-frequency pipes. The shell retains knowledge of both endpoints for cleanup.
- **Transferable ArrayBuffers (MAY):** For binary data (audio buffers, state snapshots), the pipe MAY support zero-copy ArrayBuffer transfer via the postMessage transfer list.
- **Named pipe groups:** Selective broadcast to named groups. For v1, `"*"` (all pipes) is sufficient.
- **Backpressure:** No browser API provides backpressure for postMessage. Documented as future extension.

**Security Considerations:**
- The shell is the sole broker. Napplets cannot connect directly (iframe sandbox prevents MessagePort transfer without shell mediation).
- Auth-on-open means pipe messages after PIPE_ACK are NOT individually signed. Trust is established by the AUTH handshake + `MessageEvent.source` verification. A compromised browser can forge messages on an open pipe.
- The shell identifies pipe message senders via `MessageEvent.source` (unforgeable Window reference).
- PIPE_OPEN targets by dTag. The shell MUST verify the target napplet exists and is authenticated before forwarding.
- Pipe payloads are opaque to the shell. The shell MUST NOT attempt to parse, validate, or transform pipe payload content.
- Broadcast to `"*"` reaches all open pipes. Napplets SHOULD NOT send sensitive data via broadcast.

**Implementations:**
- No implementations yet. This spec is a draft for future implementation.
