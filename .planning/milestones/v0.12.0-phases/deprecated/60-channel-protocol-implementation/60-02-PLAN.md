```
NUB-NOSTRDB

Local Event Database
```

**Description:** One paragraph explaining that NUB-NOSTRDB provides napplets with access to a local Nostr event database maintained by the shell. This enables napplets to query cached events, add events to the local store, look up events by ID, retrieve replaceable events, count matching events, and subscribe to live updates. The shell typically backs this with an OPFS-based worker relay or similar persistent event store.

**API Surface:**
```typescript
interface NostrDb {
  query(filters: NostrFilter | NostrFilter[]): Promise<NostrEvent[]>;
  add(event: NostrEvent): Promise<boolean>;
  event(id: string): Promise<NostrEvent | undefined>;
  replaceable(kind: number, author: string, identifier?: string): Promise<NostrEvent | undefined>;
  count(filters: NostrFilter | NostrFilter[]): Promise<number>;
  supports(): Promise<string[]>;
  subscribe(filters: NostrFilter | NostrFilter[]): AsyncGenerator<NostrEvent>;
}
```

Brief description of each method:
- `query()` returns events matching NIP-01 filters from the local database.
- `add()` inserts an event into the local database. Returns `true` on success.
- `event()` retrieves a single event by its ID. Returns `undefined` if not found.
- `replaceable()` retrieves the latest replaceable event (NIP-33) for a given kind, author, and optional `d` tag identifier.
- `count()` returns the number of events matching the filters (NIP-45 semantics).
- `supports()` returns the list of supported method names.
- `subscribe()` returns an AsyncGenerator that yields new events matching the filters as they arrive. The generator runs until the caller breaks out of the iteration loop, at which point the subscription is automatically cleaned up.

**Shell Behavior:**
- The shell MUST route NIPDB_REQUEST events to its local event database implementation.
- The shell MUST respond with NIPDB_RESPONSE events containing the result as JSON in the content field.
- The shell MUST support the correlation `id` tag to match responses to requests.
- The shell MAY implement `subscribe` by pushing `event-push` responses with matching `sub-id` tags when new events arrive.
- The shell MAY enforce ACL checks on database access.
- The shell SHOULD persist the database across page reloads (OPFS, IndexedDB, or equivalent).

**Event Kinds:**

| Kind | Name | Direction | Description |
|------|------|-----------|-------------|
| 29006 | NIPDB_REQUEST | napplet -> shell | Method name in `method` tag, correlation `id` tag, request payload in content |
| 29007 | NIPDB_RESPONSE | shell -> napplet | Matching `id` tag, result payload in content. For subscriptions: `method: event-push` with `sub-id` tag |

**Security Considerations:**
- The shell controls what data the napplet can see. The local database MAY contain events from multiple users; the shell MAY filter results based on the napplet's ACL.
- `add()` allows napplets to insert events. The shell SHOULD validate event signatures before storing.
- The database is shared across napplets. Events added by one napplet are visible to others with database access. This is intentional (shared cache) but the shell MAY scope access per napplet if needed.
- The `subscribe` method creates long-lived connections. The shell SHOULD clean up subscriptions when the napplet disconnects.

**Implementations:**
- [@napplet/shim](https://github.com/sandwichfarm/napplet) (napplet-side)
- [@napplet/shell](https://github.com/sandwichfarm/napplet) (shell-side)

```
NUB-IPC
=======

Inter-Napplet Pub/Sub
---------------------

`draft`

**NUB ID:** NUB-IPC
**Namespace:** `window.napplet.ipc`
**Discovery:** `shell.supports("NUB-IPC")`
```

**Description:** One paragraph explaining that NUB-IPC provides a topic-based publish/subscribe system for communication between napplets. Messages are routed through the shell using NIP-01 event subscriptions on kind 29003 (IPC_PEER). The sender publishes a signed event with a topic tag; any napplet subscribed to that topic receives the event. This is loose coupling -- the sender does not know who receives the message.

**API Surface:**
```typescript
interface NappletIpc {
  emit(topic: string, extraTags?: string[][], content?: string): void;
  on(topic: string, callback: (payload: unknown, event: NostrEvent) => void): Subscription;
}

interface Subscription {
  close(): void;
}
```

Brief description:
- `emit()` broadcasts a message to all napplets subscribed to the given topic. Creates a signed kind 29003 event with `['t', topic]` tag and posts it to the shell. `extraTags` allows additional NIP-01 tags. `content` is the event content (typically JSON).
- `on()` subscribes to messages on a topic. Under the hood, opens a `REQ` subscription filtered by `kinds: [29003], '#t': [topic]`. The callback receives `(payload, event)` where `payload` is the JSON-parsed content (or `{}` if parsing fails) and `event` is the raw NostrEvent. Returns a handle with `close()` to unsubscribe.

**Shell Behavior:**
- The shell MUST route kind 29003 events to all napplets with matching subscriptions.
- The shell MUST use NIP-01 filter matching on the `#t` (topic) tag for subscription routing.
- The shell MUST NOT deliver an event back to the sender (sender exclusion).
- The shell MAY enforce ACL checks on IPC capabilities.
- The shell MUST support standard NIP-01 subscription lifecycle (REQ/EVENT/EOSE/CLOSE) for IPC subscriptions.

**Topic Conventions:** Topics use a prefix convention to signal direction:

| Prefix | Direction | Meaning |
|--------|-----------|---------|
| `shell:*` | napplet -> shell | Commands sent by a napplet to the shell |
| `napplet:*` | shell -> napplet | Responses/notifications from shell to napplet |
| `{domain}:*` | bidirectional | Domain-scoped messages (e.g., `auth:*`, `chat:*`, `audio:*`) |

These conventions are advisory. The shell routes by filter match, not by prefix.

**Event Kinds:**

| Kind | Name | Direction | Description |
|------|------|-----------|-------------|
| 29003 | IPC_PEER | bidirectional | Topic in `t` tag, payload in content. Shell routes by subscription filter. |

**Security Considerations:**
- IPC messages are signed with the napplet's delegated session key. The shell verifies signatures before routing.
- Topic namespaces are not enforced -- any napplet can emit on any topic. The shell MAY restrict topics via ACL.
- Content is JSON-serializable strings. The receiving napplet is responsible for validating parsed content.
- IPC is per-message authenticated (full NIP-01 event with signature). For high-frequency communication where per-message signing overhead is unacceptable, use NUB-PIPES instead.

**Relationship to NUB-PIPES:** NUB-IPC is for loose-coupled, topic-based pub/sub with per-message authentication. NUB-PIPES is for tight-coupled, point-to-point connections with auth-on-open. Both coexist -- IPC for infrequent coordination, pipes for data streams.

**Implementations:**
- [@napplet/shim](https://github.com/sandwichfarm/napplet) (napplet-side)
- [@napplet/shell](https://github.com/sandwichfarm/napplet) (shell-side)
