## Objective

Write the complete NIP-5D specification document: "Nostr Web Applets" -- a protocol for sandboxed iframe applications to communicate with a hosting shell via postMessage using NIP-01 wire format. The NIP covers transport, AUTH handshake, relay proxy, capability discovery, six standard capabilities, MUST/MAY layering, and security considerations. Under 500 lines, terse, in nostr-protocol/nips markdown format.

## Context

NIP-5D is the runtime communication protocol layer in a three-layer model:
- NIP-5A = hosting (static websites/nsites)
- NIP-5B = discovery (app listings)  
- NIP-5D = runtime communication (sandboxed app <-> shell protocol)

The source material is SPEC.md (1520 lines, 41KB). This plan distills it to a standards-track NIP under 500 lines. The NIP defines observable wire protocol only -- no internal implementation details.

<action>
Add `## Authentication` section. This is the core of the protocol (~80-100 lines).

Start with ASCII sequence diagram:

```
Napplet                           Shell
  |                                 |
  |-- REGISTER {dTag, hash} ------->|
  |                                 |  derive keypair
  |<-- IDENTITY {pub, priv} --------|
  |<-- AUTH <challenge> ------------|
  |                                 |
  |-- AUTH <signed kind 22242> ---->|
  |                                 |  verify signature
  |<-- OK <event_id, true, ""> ----|
  |                                 |
```

Then define each step in prose:

1. **REGISTER**: Napplet sends `["REGISTER", {"dTag": "<type>", "claimedHash": "<hash>"}]` as first message. `dTag` from `<meta name="napplet-type">`, `claimedHash` from `<meta name="napplet-aggregate-hash">` or empty string.

2. **IDENTITY**: Shell derives keypair via `HMAC-SHA256(shellSecret, dTag + aggregateHash)` and sends `["IDENTITY", {"pubkey": "<hex>", "privkey": "<hex>", "dTag": "<type>", "aggregateHash": "<hash>"}]`. Shell secret is a 32-byte random value persisted per shell instance.

3. **AUTH challenge**: Shell sends `["AUTH", "<uuid-challenge>"]` per NIP-42 pattern.

4. **AUTH response**: Napplet signs kind 22242 event per NIP-42 with additional tags: `["type", "<dTag>"]`, `["version", "2.0.0"]`, `["aggregateHash", "<hash>"]`. Relay tag MUST be `"napplet://shell"`.

5. **Verification**: Shell MUST verify kind is 22242, challenge matches, created_at within 60 seconds, Schnorr signature valid, pubkey matches delegated key.

6. **Post-AUTH**: Shell identifies senders by MessageEvent.source Window reference. No per-message signature verification (verify once, trust source).

7. **Pre-AUTH queueing**: Messages before AUTH MUST be queued and replayed on success, dropped on failure. Queue cap: 50 messages.

8. **Manifest check**: After AUTH, shell checks napplet's `requires` tags from NIP-5A manifest against available services. Shell MAY reject if requirements unmet.
</action>

The shell acts as a NIP-01 relay to the napplet. Define:

1. **Subscription handling**: REQ filters per NIP-01. Shell delivers matching events from connected relays and local cache. EOSE when stored events exhausted.

2. **Event publishing**: Napplet sends EVENT, shell forwards to connected relays. Shell MUST verify AUTH is complete. Shell responds with OK.

5. **Bus kinds**: Kinds 29000-29999 are postMessage bus traffic, never forwarded to external relays. Standard kinds are forwarded.

<action>
Add `## Security Considerations` section (~60-80 lines).

**Threat model**: Napplets are untrusted code. The shell is trusted. The browser enforces iframe sandbox boundaries.

**Mitigations** (numbered list):

1. **Iframe sandbox**: `allow-scripts` without `allow-same-origin`. Napplets have opaque origins, no direct state access, no direct network access beyond fetch to serving origin. Note: `allow-scripts` + `allow-same-origin` together would allow sandbox escape.

2. **postMessage `*` origin**: Required because sandboxed iframes have opaque ("null") origins. Sender authentication uses `MessageEvent.source` (unforgeable Window reference), NOT `event.origin`. This is not a security weakness -- it is the correct approach for sandboxed iframes.

3. **AUTH handshake**: One-time Schnorr signature verification establishes napplet identity. After AUTH, Window reference identity is sufficient. This "verify once, trust source" model avoids per-message cryptographic overhead.

4. **Delegated key confinement**: Shell-derived keypairs are confined to the postMessage channel. Events signed with delegated keys MUST NOT be published to external relays. The user's signer (NIP-07/NIP-46) is the only source of relay-published events.

5. **Storage isolation**: State is scoped by composite key `(dTag, aggregateHash)`. Napplets cannot access each other's data. Hash changes create new namespaces.

6. **Aggregate hash verification**: Shell verifies napplet build integrity against NIP-5A manifests. Hash mismatch results in AUTH rejection.

**Non-guarantees** (the protocol does NOT protect against):
- A compromised browser
- A malicious shell implementation (napplets must trust the shell)
- Side-channel attacks through timing or resource usage
- Social engineering the user into granting capabilities
</action>
