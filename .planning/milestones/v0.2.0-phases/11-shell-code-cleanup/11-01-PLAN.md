# Plan 11-01: Method Naming, API Surface, and Internal Visibility Cleanup

<objective>
Ensure all public ShellBridge methods follow verb-noun naming convention, minimize the public API surface exported from @napplet/shell, and make internal helpers truly private (unexported). After this plan, the package's public API is clean, consistent, and minimal.
</objective>

<context>
By the time this plan executes, Phases 7-10 have completed:
- Phase 7 renamed pseudo-relay.ts to shell-bridge.ts, PseudoRelay to ShellBridge, createPseudoRelay to createShellBridge
- Phase 8 extracted ACL logic to @napplet/acl package, acl-store.ts became a thin persistence adapter
- Phase 9 added a single enforce() gate in ShellBridge
- Phase 10 added exhaustive behavioral tests

The cleanup targets the post-Phase-10 codebase. File paths reference the renamed files.
</context>

<tasks>

<task id="11-01-01" title="Audit and rename inconsistent ShellBridge public methods">
<read_first>
- packages/shell/src/shell-bridge.ts
- packages/shell/src/index.ts
</read_first>
<action>
Review every method on the ShellBridge interface and the createShellBridge factory function. Apply verb-noun naming convention consistently:

1. If `onConsentNeeded(handler)` still exists, rename to `registerConsentHandler(handler)` — the `on*` pattern is for events, but this is a registration function that accepts a callback. Update both the interface definition and the implementation.

2. If `cleanup()` is still a bare verb, rename to `destroy()` — standard lifecycle method name that clearly communicates the instance is no longer usable after calling.

3. If `deliverToSubscriptions(event, senderId)` is an internal helper, confirm it is NOT on the public interface. If it leaked to the public interface, remove it.

4. If `storeAndRoute(event, senderId)` is an internal helper, rename to `bufferAndDeliver(event, senderId)` — clearer about what it does (buffers in ring buffer, delivers to subscriptions).

5. Verify all internal handler functions follow the pattern: `handleEvent`, `handleReq`, `handleClose`, `handleCount`, `handleAuth`, `handleSignerRequest`, `handleHotkeyForward`, `handleAudioCommand`, `handleShellCommand`. These should already be verb-noun.

Update the ShellBridge interface type to reflect any renamed public methods.
</action>
<acceptance_criteria>
- `grep -n 'onConsentNeeded' packages/shell/src/shell-bridge.ts` returns 0 results (renamed)
- `grep -n 'registerConsentHandler' packages/shell/src/shell-bridge.ts` returns at least 2 results (interface + implementation)
- `grep -n 'cleanup()' packages/shell/src/shell-bridge.ts` returns 0 results for the public method (renamed to destroy)
- `grep -n 'destroy()' packages/shell/src/shell-bridge.ts` returns at least 2 results (interface + implementation)
- `grep -n 'storeAndRoute' packages/shell/src/shell-bridge.ts` returns 0 results (renamed)
- Every method on the ShellBridge interface follows verb-noun or verb pattern: handleMessage, sendChallenge, injectEvent, destroy, registerConsentHandler
- `pnpm build` succeeds
</acceptance_criteria>
</task>

<task id="11-01-02" title="Minimize and audit public API surface in index.ts">
<read_first>
- packages/shell/src/index.ts
- packages/shell/src/shell-bridge.ts
- packages/shell/src/acl-store.ts
- packages/shell/src/state-proxy.ts
- apps/demo/src/shell-host.ts
</read_first>
<action>
Review every export in index.ts and determine if it belongs in the public API:

1. **Keep as public exports** (used by shell host implementations):
   - `createShellBridge` and `ShellBridge` type — primary API
   - All `ShellHooks` and sub-hook interfaces — needed by host apps
   - `originRegistry` — host registers iframe windows
   - `nappKeyRegistry` — host checks napp registration status
   - `audioManager` — host builds audio UI
   - `manifestCache` — host manages manifest lifecycle
   - `cleanupNappState` — host calls on window close
   - `BusKind`, `AUTH_KIND`, `SHELL_BRIDGE_URI`, `PROTOCOL_VERSION` — protocol constants
   - `ALL_CAPABILITIES` — host configures ACL
   - `TOPICS` — host uses topic constants
   - Type exports: `NostrEvent`, `NostrFilter`, `NappKeyEntry`, `AclEntry`, `Capability`, `ConsentRequest`, `AudioSource`, `ManifestCacheEntry`, `PendingUpdate`, `TopicKey`, `TopicValue`

2. **Remove from public exports** (internal implementation details):
   - `handleStateRequest` — only called internally by ShellBridge's enforce gate (after Phase 9). If demo or tests import it directly, update them to go through ShellBridge instead. If still needed externally, keep but document why.
   - `DEFAULT_STATE_QUOTA` — internal constant, not needed by host apps. If used externally, keep.
   - `DESTRUCTIVE_KINDS` — internal concern (consent flow). Should be handled by ShellBridge internally.
   - `aclStore` — after Phase 8, this is a thin persistence adapter. If host needs ACL operations, they should go through @napplet/acl directly or through ShellBridge methods.

3. For each removal, check `apps/demo/src/shell-host.ts` and test files for direct imports. If they import the removed symbol, update to use the public API path instead.

4. Add a `// Public API` section comment at top of index.ts and an `// Internal re-exports (used by tests/demo)` section if some exports must stay for testing but aren't part of the intended public API.
</action>
<acceptance_criteria>
- `packages/shell/src/index.ts` has clear section comments separating public API from internal/testing exports
- If `handleStateRequest` was removed: `grep 'handleStateRequest' packages/shell/src/index.ts` returns 0 results
- If `DESTRUCTIVE_KINDS` was removed from index.ts: verify it's not imported from `@napplet/shell` anywhere except internal files
- `pnpm build` succeeds
- `pnpm type-check` succeeds
</acceptance_criteria>
</task>

<task id="11-01-03" title="Remove export keyword from internal-only module helpers">
<read_first>
- packages/shell/src/acl-store.ts
- packages/shell/src/napp-key-registry.ts
- packages/shell/src/state-proxy.ts
- packages/shell/src/origin-registry.ts
- packages/shell/src/manifest-cache.ts
</read_first>
<action>
For each module file in packages/shell/src/, audit every `export` keyword:

1. **acl-store.ts:**
   - `aclKey()` — used only internally within the file. Remove `export` keyword if not imported by any other file in the package. Check with: `grep -rn 'aclKey' packages/shell/src/ --include='*.ts'` — if only acl-store.ts references it, remove export.
   - `DESTRUCTIVE_KINDS` re-export — if removed from index.ts in task 11-01-02, also remove the `export { DESTRUCTIVE_KINDS }` re-export line from acl-store.ts.

2. **napp-key-registry.ts:**
   - `getPendingUpdateVersion()` — check if imported outside the file. If only used internally or by demo, decide whether to keep. If kept, document why. If not needed externally, remove export.

3. **state-proxy.ts:**
   - `handleStateRequest` — if removed from index.ts, check if it still needs `export` (it's imported by shell-bridge.ts, so it needs `export` at the module level even if not re-exported from index.ts).
   - `cleanupNappState` — same analysis.

4. **For each file:** Run `grep -rn 'import.*from.*./filename' packages/shell/src/` to identify cross-module imports. Only keep `export` on symbols that are imported by other files within the package OR re-exported from index.ts.

The goal is: if a symbol is only used within its own file, remove the `export` keyword. If it's used by other files in the package but not by external consumers, keep `export` but don't re-export from index.ts.
</action>
<acceptance_criteria>
- `aclKey` function in acl-store.ts: if only used within acl-store.ts, `grep -n '^export function aclKey' packages/shell/src/acl-store.ts` returns 0 results (changed to `function aclKey`)
- Every `export` in module files maps to either: (a) re-exported from index.ts, or (b) imported by another file in packages/shell/src/
- `pnpm build` succeeds
- `pnpm type-check` succeeds
</acceptance_criteria>
</task>
