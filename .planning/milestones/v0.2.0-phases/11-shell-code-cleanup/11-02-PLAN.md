# Plan 11-02: JSDoc Documentation, Debug Removal, and Error Handling Cleanup

<objective>
Add comprehensive JSDoc with @param, @returns, and @example to all exported functions and types. Remove all debug console.log statements. Ensure every catch block has a clear comment explaining the error handling strategy. After this plan, the shell package is fully documented with no development artifacts.
</objective>

<context>
By the time this plan executes, Phases 7-10 have completed and Plan 11-01 may have run in parallel (same wave). This plan focuses on documentation and error handling — concerns orthogonal to the naming/API changes in 11-01. If both run in wave 1, they target different aspects of the same files. If ordering is enforced, this plan should run after 11-01 since it documents the final API names.
</context>

<task id="11-02-02" title="Add explanatory comments to all silent catch blocks">
<read_first>

- packages/shell/src/manifest-cache.ts
- packages/shell/src/state-proxy.ts
</read_first>
<action>
Audit every `catch` block in the shell package. Every catch block must have a comment explaining WHY the error is handled that way. Apply these specific fixes:

1. **manifest-cache.ts line 46** — `catch { cache.clear(); }`
   Change to: `catch { /* Corrupted cache data — clear and start fresh */ cache.clear(); }`

2. **manifest-cache.ts line 52** — `catch { /* silent */ }`
   Change to: `catch { /* localStorage unavailable in sandboxed contexts — persist is best-effort */ }`

3. **manifest-cache.ts line 57** — `catch { /* ignore */ }`
   Change to: `catch { /* localStorage unavailable — cleanup is best-effort */ }`

4. **shell-bridge.ts** (was pseudo-relay.ts) — all catch blocks. Expected catch blocks and their comments:
   - Hotkey forward catch: `catch { /* Best-effort hotkey forwarding — napp-reported key data may be malformed */ }`
   - Worker relay event cache catch: `catch { /* Worker relay cache write is best-effort — does not affect subscription delivery */ }`
   - JSON parse catches (event tag, content, filters): These already send error responses (`sendOk(false, 'error: ...')`) — these are proper error handling, just ensure the catch has a brief comment like `/* Invalid JSON from napp */`
   - DM content parse catch: `catch { /* Malformed DM content from napp */ }`

5. **acl-store.ts line 122-124** — `catch { // localStorage unavailable }`
   This comment is adequate. Keep as-is.

6. **acl-store.ts line 154-155** — `catch { store.clear(); }`
   Change to: `catch { /* Corrupted ACL data in localStorage — clear and use defaults */ store.clear(); }`

7. **acl-store.ts line 168-170** — `catch { // Ignore }`
   Change to: `catch { /* localStorage unavailable — ACL clear is best-effort */ }`

8. **state-proxy.ts line 97-98** — `catch { sendError(...) }`
   This is proper error handling (sends error response). Add comment: `catch { /* localStorage write failed (quota exceeded or unavailable) */ sendError(...) }`

9. **Any new catch blocks added by Phases 7-10** — audit and add explanatory comments.

Rule: No catch block should have an empty body or just `{ }`. Every catch must either (a) re-throw, (b) send an error response, or (c) swallow with a comment explaining the best-effort strategy.
</action>

<task id="11-02-03" title="Add JSDoc to ShellBridge interface and createShellBridge">
<read_first>
- packages/shell/src/shell-bridge.ts
</read_first>
<action>
Add comprehensive JSDoc with @param, @returns, and @example to the ShellBridge interface and createShellBridge factory:

1. **ShellBridge interface** — add JSDoc to the interface itself and each method:

```typescript
/**
 * Shell-side message bridge that handles NIP-01 communication with napplet iframes.
 *
 * @example
 * ```ts
 * import { createShellBridge } from '@napplet/shell';
 *
 * const bridge = createShellBridge(hooks);
 * window.addEventListener('message', bridge.handleMessage);
 * bridge.sendChallenge('window-1');
 * ```
 */
export interface ShellBridge {
  /**
   * Handle an incoming postMessage from a napplet iframe.
   * Dispatches to the appropriate verb handler (EVENT, REQ, CLOSE, COUNT, AUTH).
   *
   * @param event - The raw MessageEvent from window.addEventListener('message', ...)
   * @example
   * ```ts
   * window.addEventListener('message', bridge.handleMessage);
   * ```
   */
  handleMessage(event: MessageEvent): void;

  /**
   * Send a NIP-42 AUTH challenge to a napplet window, initiating the handshake.
   *
   * @param windowId - The window identifier registered via originRegistry
   * @example
   * ```ts
   * bridge.sendChallenge('napp-window-1');
   * ```
   */
  sendChallenge(windowId: string): void;

  /**
   * Inject a shell-originated event into subscription delivery.
   * Used for broadcasting shell state changes (e.g., auth identity) to napplets.
   *
   * @param topic - The event topic tag value (e.g., 'auth:identity-changed')
   * @param payload - The event content, will be JSON.stringify'd
   * @example
   * ```ts
   * bridge.injectEvent('auth:identity-changed', { pubkey: userPubkey });
   * ```
   */
  injectEvent(topic: string, payload: unknown): void;

  /**
   * Destroy the bridge instance, cleaning up all internal state.
   * Call when the shell is shutting down or the bridge is no longer needed.
   *
   * @example
   * ```ts
   * bridge.destroy();
   * ```
   */
  destroy(): void;

  /**
   * Register a handler for consent requests on destructive signing kinds.
   * Called when a napplet requests signing for kinds 0, 3, 5, or 10002.
   *
   * @param handler - Callback receiving the consent request with a resolve function
   * @example
   * ```ts
   * bridge.registerConsentHandler((request) => {
   *   const allowed = confirm(`Allow ${request.pubkey} to sign kind ${request.event.kind}?`);
   *   request.resolve(allowed);
   * });
   * ```
   */
  registerConsentHandler(handler: (request: ConsentRequest) => void): void;
}
```

2. **createShellBridge factory** — update existing JSDoc to include @example:

```typescript
/**
 * Create a ShellBridge instance with dependency injection via hooks.
 *
 * @param hooks - Host application provides relay pool, auth, config, etc.
 * @returns A ShellBridge instance ready to handle napp messages
 * @example
 * ```ts
 * import { createShellBridge, type ShellHooks } from '@napplet/shell';
 *
 * const hooks: ShellHooks = {
 *   relayPool: myRelayPoolHooks,
 *   relayConfig: myRelayConfigHooks,
 *   windowManager: myWindowManagerHooks,
 *   auth: myAuthHooks,
 *   config: myConfigHooks,
 *   hotkeys: myHotkeyHooks,
 *   workerRelay: myWorkerRelayHooks,
 *   crypto: myCryptoHooks,
 * };
 * const bridge = createShellBridge(hooks);
 * ```
 */
```

Note: Method names used above (destroy, registerConsentHandler) reflect the renames from Plan 11-01. If Plan 11-01 hasn't run yet or used different names, adjust to match the actual current names.
</action>
