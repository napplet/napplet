# Plan 31-01: Signer Connection State Model and NIP-07 Connect Flow

## Objective

Replace the hidden mock signer (`signer-demo.ts`) as the primary user-facing experience. Introduce a `SignerConnectionState` model, implement the NIP-07 connect path, and update the signer service node in the topology to reflect connected/disconnected state.

After this plan:
- The demo boots with `getSigner()` returning `null` — no hidden mock signer.
- A "Not connected" state is visible on the signer node.
- Users with a NIP-07 browser extension can click to connect and see their pubkey reflected.
- The signer service responds with `error: no signer configured` for requests before connection (correct, visible behavior).

## Tasks

### Task 1: Create `apps/demo/src/signer-connection.ts`

Introduce a module that owns the live signer connection state:

```ts
export type SignerConnectionMethod = 'nip07' | 'nip46' | 'none';

export interface SignerRequestRecord {
  timestamp: number;
  method: string;       // 'signEvent', 'getPublicKey', etc.
  kind?: number;        // for signEvent requests
  success: boolean;
}

export interface SignerConnectionState {
  method: SignerConnectionMethod;
  pubkey: string | null;
  relay: string | null;          // NIP-46 only; null for NIP-07
  recentRequests: SignerRequestRecord[];
  isConnecting: boolean;
  error: string | null;
}
```

The module maintains a single mutable `_state: SignerConnectionState` initialized as:
```ts
{ method: 'none', pubkey: null, relay: null, recentRequests: [], isConnecting: false, error: null }
```

Exported API:
- `getSignerConnectionState(): SignerConnectionState` — returns a shallow copy of current state
- `getSigner(): RuntimeSigner | null` — returns the active signer or `null`
- `recordSignerRequest(record: SignerRequestRecord): void` — appends to `recentRequests` (max 20, oldest dropped)
- `connectNip07(): Promise<void>` — NIP-07 connect flow (see Task 2)
- `disconnectSigner(): void` — resets to none/null state
- `onStateChange(cb: (state: SignerConnectionState) => void): () => void` — subscribe to state updates

The `getSigner()` function reads a module-level `_activeSigner: RuntimeSigner | null` reference, not the state object, so callers get a direct reference suitable for passing to the signer service.

### Task 2: Implement `connectNip07()` in `signer-connection.ts`

```ts
export async function connectNip07(): Promise<void> {
  const nostr = (window as unknown as Record<string, unknown>).nostr as Nip07Signer | undefined;
  if (!nostr || typeof nostr.getPublicKey !== 'function') {
    _setState({ ..._state, error: 'No NIP-07 extension detected', isConnecting: false });
    return;
  }
  _setState({ ..._state, isConnecting: true, error: null });
  try {
    const pubkey = await nostr.getPublicKey();
    _activeSigner = buildNip07Adapter(nostr);
    _setState({ ..._state, method: 'nip07', pubkey, relay: null, isConnecting: false, error: null });
  } catch (err) {
    _setState({ ..._state, isConnecting: false, error: `Connection failed: ${(err as Error).message}` });
  }
}
```

`buildNip07Adapter(nostr)` wraps `window.nostr` into a `RuntimeSigner`-compatible object. The `RuntimeSigner` interface (from `packages/runtime/src/types.ts`) requires `getPublicKey()`, `signEvent()`, and optionally `nip04`, `nip44`, `getRelays`.

The adapter should forward all available methods from the extension and return `null`/undefined for unavailable optional methods.

### Task 5: Wire Signer State Updates to the Topology Node in `main.ts`

In `main.ts`, subscribe to signer connection state changes and update the rendered signer node:

```ts
import { onStateChange, connectNip07, getDemoSignerState } from './signer-connection.js';
import { getServiceNodeId } from './topology.js';

const signerNodeId = getServiceNodeId('signer');

onStateChange((state) => {
  const signerNode = document.getElementById(signerNodeId);
  if (!signerNode) return;
  updateSignerNodeDisplay(signerNode, state);
});

document.addEventListener('click', (e) => {
  const target = e.target as HTMLElement;
  if (target.closest('[data-action="open-signer-connect"]')) {
    connectNip07(); // Temporary: NIP-07 direct connect; modal comes in Plan 31-02
  }
});
```

The `updateSignerNodeDisplay()` function reads the state and sets appropriate CSS classes and text content on the signer node element without re-rendering the entire topology.

### Task 6: Update the Debugger on Signer Connection

When a signer connection state change occurs, log to the debugger:

```ts
onStateChange((state) => {
  if (state.method !== 'none' && !state.isConnecting && !state.error) {
    debuggerEl?.addSystemMessage(
      `signer connected via ${state.method}: ${state.pubkey?.substring(0, 16)}...`
    );
  }
  if (state.error) {
    debuggerEl?.addSystemMessage(`signer connection error: ${state.error}`);
  }
});
```

### Task 7: Update `getDemoHostPubkey` Export

`main.ts` currently calls `getDemoHostPubkey()` to display the shell pubkey on the shell node. After removing the mock signer import, this export should come from `getSignerConnectionState().pubkey` or from a dedicated host identity separate from the signer. Because the shell node's "host pubkey" concept is distinct from the signer's identity, introduce a minimal `host-identity.ts` module that generates and stores a static ephemeral host keypair for the shell node display. This separates shell identity from signer identity.

Alternatively, re-export a `getDemoHostPubkey()` shim from `shell-host.ts` that returns the connected signer's pubkey if available, or a static placeholder string. The simplest correct approach for this plan is the static placeholder, with Phase 31-03 able to refine it.

## Acceptance Criteria

1. `pnpm --filter @napplet/demo build` succeeds with no TypeScript errors.

3. The signer node in the topology renders a "Not connected" state on boot.

5. Without a NIP-07 extension, clicking connect shows an error state on the signer node.
6. The signer service responds with `error: no signer configured` for signer requests before connection.
7. The debugger logs signer connection events and errors.
