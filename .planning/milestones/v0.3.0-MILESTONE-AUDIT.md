## Cross-Phase Integration

### Wired (No Issues)

- Core → Runtime: types and constants imported correctly
- Core → Shell: re-exports are reference-equal (same objects)
- Core → Shim: thin re-exports via types.ts
- Runtime → Shell: createShellBridge delegates to createRuntime(adaptHooks(hooks))
- ACL → Runtime: acl-state.ts wraps @napplet/acl with RuntimeAclPersistence
- All AUTH paths require completed handshake before processing

### Tech Debt Items

1. **Dead code exports (SHELL-06):** `handleStateRequest` and `cleanupNappState` re-exported from old `state-proxy.ts` in shell's `index.ts`. These are browser-coupled and reference the empty shell singleton `nappKeyRegistry`. Any external integrator calling them would get silent failures.

2. **Duplicate enforce.ts (SHELL-04):** Shell has its own copy of `enforce.ts` rather than re-exporting from `@napplet/runtime`. Deliberate decision per 14-01 SUMMARY. Shell's copy is untested by integration tests.

3. **SVC-01 scope (Phase 15):** `services` field is on `ShellHooks`, not `RuntimeHooks`. Correct for v0.3.0 design-only scope. Needs migration to `RuntimeHooks` when v0.4.0 implements services.

4. **Shell singletons:** `nappKeyRegistry` and `aclStore` exported from shell but empty at runtime — registrations go to runtime's internal instances. The e2e harness correctly uses `relay.runtime.nappKeyRegistry`.
