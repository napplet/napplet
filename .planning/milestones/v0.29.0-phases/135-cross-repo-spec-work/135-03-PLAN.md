**## Origin Format**

Dedicated rules, presented as a bulleted MUST list. Each rule cites the rationale from research/pitfalls:

- **Scheme whitelist**: The scheme part of an origin MUST be one of `https:`, `wss:`, `http:`, or `ws:`. Any other scheme (including `file:`, `ftp:`, `gopher:`, `data:`, or custom schemes) MUST be rejected at manifest-load time by the shell and at build time by the napplet build tool.
- **Lowercase scheme + host**: The scheme and host parts MUST be lowercase. `HTTPS://FOO.COM` is invalid on the wire. Build tools MUST lowercase before emitting; shells MUST reject non-lowercase input (shells do not auto-lowercase on lookup; see the aggregateHash determinism rationale below).
- **Punycode IDN**: Hosts containing non-ASCII characters MUST be Punycode-encoded (`xn--` form) before emission. The build tool is the sole authority for UTF-8 → Punycode conversion. The shell MUST NOT auto-convert; the shell MUST reject any non-Punycode non-ASCII host with a diagnostic.
- **No wildcards**: The host MUST NOT contain `*`. Wildcard subdomains (e.g., `https://*.example.com`) MUST be rejected at build and shell. Each subdomain is a separate origin requiring a separate tag.
- **Default ports banned**: If the port component would be the scheme's default (443 for `https`/`wss`, 80 for `http`/`ws`), the port MUST be omitted from the origin. `https://foo.com:443` MUST be rejected at build; `https://foo.com` is the correct form. The rationale is aggregateHash determinism (two cosmetically different forms that CSP treats as equivalent would otherwise produce different hashes). Shells MUST NOT normalize default ports on lookup.
- **No path / query / fragment**: The origin is scheme + host + optional non-default port. `https://foo.com/api` is invalid; `https://foo.com?x=1` is invalid; `https://foo.com#section` is invalid. CSP `connect-src` matches origins only; path-level constraints are out of scope.
- **Trailing slash omitted**: The origin MUST NOT end in `/`. `https://foo.com/` is invalid; `https://foo.com` is the correct form.

**## Canonical `connect:origins` aggregateHash Fold**

Normative pseudocode block (SPEC-02):

```
# Input: origins = list[string] (already scheme-lowercased, host-lowercased, Punycode-encoded,
#                                 no-path, no-trailing-slash, no-default-port — i.e., post-normalization)
# Output: a 64-character lowercase hex SHA-256 digest

def connect_origins_fold(origins: list[str]) -> str:
    normalized = [o.lower() for o in origins]        # step 1: lowercase (idempotent after normalization)
    normalized.sort()                                 # step 2: ASCII-ascending sort
    joined = "\n".join(normalized)                    # step 3: join with single LF, NO trailing newline
    encoded = joined.encode("utf-8")                  # step 4: UTF-8 encode
    digest = sha256(encoded).hexdigest()              # step 5: SHA-256
    return digest.lower()                             # step 6: lowercase hex (already lowercase from hexdigest)
```

After the pseudocode, prose: "The result is pushed as a synthetic xTag entry `[<hex-digest>, 'connect:origins']` into the manifest's xTag array BEFORE aggregateHash is computed. This synthetic entry participates in aggregateHash but is filtered out of the `['x', ...]` projection in the signed manifest. Shells performing grant lookups MUST recompute this fold from the manifest's `['connect', ...]` tags and verify it matches the synthetic `connect:origins` xTag's hash — mismatch MUST result in manifest rejection. This double-entry (origin tags + folded hash) is the guard against silent origin-list tampering post-signing."

**## Grant Persistence**

Short section:

**## Security Considerations**

Three subsections:

- **Post-grant opacity**: Once origins are approved, the shell has zero browser-level hook to inspect or filter subsequent network traffic between the napplet and the approved origins. This is a fundamental tradeoff. Consent-prompt language MUST reflect this (see **Shell Consent Flow**). Audit logging, per-request quotas, and traffic filtering all require a service worker, which the napplet's `sandbox="allow-scripts"` iframe forbids from registering.
- **Mixed-content silent failure**: Browsers block `http:` and `ws:` fetches from napplets running in shells served over `https:`, regardless of CSP. A napplet declaring `http:` origins approved by the user will silently fail to fetch when the shell is on `https:`, except for the localhost / `127.0.0.1` secure-context exception. Napplet build tools SHOULD emit a build-time warning on cleartext origins; shells SHOULD surface this reality in the consent prompt.
- **Sandbox preservation**: Approving `connect` origins does NOT relax the iframe sandbox. The napplet remains at an opaque origin with `sandbox="allow-scripts"`. Grants relax only the `connect-src` CSP directive — other browser protections (Same-Origin Policy against the shell, absence of cookies, no service-worker registration, no shared-storage access) remain in force. A grant is a network-egress permission, not a sandbox escape.

**## Graceful Degradation**

A 4-state priority ladder napplets SHOULD branch on:
1. `shell.supports('nub:connect') === true` AND `window.napplet.connect.granted === true` → use direct `fetch` / `WebSocket` / `SSE` to the approved origins
2. `shell.supports('nub:connect') === true` AND `window.napplet.connect.granted === false` → user denied; fall back to NUB-RESOURCE where possible, or degrade feature gracefully, OR display a napplet-side affordance prompting the user to re-approve through the shell's revocation UI
3. `shell.supports('nub:connect') === false` AND `shell.supports('nub:resource') === true` → shell does not implement NUB-CONNECT; use NUB-RESOURCE for read-only bytes; streaming / POST / WebSocket features are unavailable; degrade gracefully
4. Neither NUB implemented → napplet cannot reach the network; display appropriate UX and avoid silent failure
