# Summary: Plan 22.1-02 — Relay Pool and Cache Services

## What Was Done

Created two standalone ServiceHandler implementations in `@napplet/services`:

1. `createRelayPoolService()` — wraps an existing relay pool (subscribe, publish, selectRelayTier, isAvailable) as a ServiceHandler. Handles REQ (subscribe with 15s EOSE fallback), CLOSE (unsubscribe + cleanup), and EVENT (publish). Tracks subscriptions per `windowId:subId`. Cleans up all window subscriptions on `onWindowDestroyed`.

2. `createCacheService()` — wraps an existing cache (query, store, isAvailable) as a ServiceHandler. REQ is a one-shot query: delivers cached events then sends EOSE. EVENT stores in cache (best-effort). `onWindowDestroyed` is a no-op (no per-window state).

Both exported from `packages/services/src/index.ts`.

## Artifacts Created

- `packages/services/src/relay-pool-service.ts` — ServiceHandler wrapping relay pool
- `packages/services/src/cache-service.ts` — ServiceHandler wrapping cache

## Key Decisions

- Relay pool service tracks subscriptions with `Map<windowId:subId, TrackedSubscription>` — same pattern as the runtime's internal tracking but self-contained
- Cache service is stateless (one-shot queries) — no subscription tracking needed
- Both services use consistent error handling (EOSE on failure, best-effort writes)
