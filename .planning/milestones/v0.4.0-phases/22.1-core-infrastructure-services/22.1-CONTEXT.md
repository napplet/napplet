# Phase 22.1: Core Infrastructure Services - Context

**Gathered:** 2026-03-31
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate relay pool, cache, and signer from hardcoded RuntimeHooks interfaces to registered services. After this phase, all capabilities are discoverable and pluggable — the runtime is a generic message router, and shell implementors choose their own relay/cache/signer implementations.

</domain>

<decisions>
## Implementation Decisions

### Migration strategy
- **D-01:** Incremental migration. Keep existing RuntimeHooks.relayPool, .cache, and signer handling working. Add service dispatch alongside. Once service-based routing works, swap internals to delegate to services. Then remove old hook interfaces. Existing 180 tests stay green throughout.
- **D-02:** The dual path is temporary — a migration bridge, not a permanent architecture. Follow-up cleanup removes the old RuntimeHooks interfaces once service versions are verified.

### Signer as service
- **D-03:** Signer fits the handleMessage interface naturally. Napplet sends `['EVENT', { kind: 29001, ... }]`, signer service handles via handleMessage, responds with `['EVENT', subId, { kind: 29002, ... }]` via send. Existing `handleSignerRequest()` becomes the signer service implementation.

### Subscription coordination
- **D-04:** Shell coordinates, not the runtime. Shell creates a composite service (e.g., `createCoordinatedRelay(pool, cache)`) that wraps both relay pool and cache, handles dedup and unified EOSE internally. Registers as a single service. Runtime just routes REQs to it.
- **D-05:** `createCoordinatedRelay()` lives in @napplet/services as a convenience helper. Shell implementors who want custom behavior write their own composite service.
- **D-06:** This follows the "runtime is a dumb router" principle — the runtime doesn't embed relay-pool-specific knowledge (merging, dedup, EOSE timing).

### What moves, what stays
- **D-07:** Moves to services: relay pool handling, cache handling, signer handling. These become ServiceHandler implementations.
- **D-08:** Stays in runtime: AUTH handshake (identity, not a capability), ACL enforcement (filter, not a service), subscription tracking infrastructure (eventBuffer), replay detection, message routing core.
- **D-09:** RuntimeHooks loses: `relayPool`, `cache` interfaces. RuntimeHooks keeps: `auth`, `crypto`, `aclPersistence`, `manifestPersistence`, `statePersistence`, `windowManager`, `relayConfig`, `hotkeys`, `dm`, `sendToNapplet`, callbacks.

### Claude's Discretion
- Exact ordering of incremental migration steps
- Whether to introduce an adapter layer that makes old-style hooks look like services during transition
- Test migration strategy (update existing tests vs write new service-based tests)
- Whether hotkey forwarding also becomes a service or stays as runtime core

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `handleSignerRequest()` in runtime.ts — extract as-is into a signer ServiceHandler
- `RuntimeRelayPoolHooks` / `RuntimeCacheHooks` interfaces — inform the service handler design
- `createAudioService()` pattern from Phase 20 — template for createRelayPoolService(), createCacheService(), createSignerService()
- `createCoordinatedRelay()` — new helper combining relay + cache services

### Established Patterns
- RuntimeHooks DI — services continue this pattern at a different level
- eventBuffer subscription tracking — shared infrastructure that services use via send() callback
- Incremental migration pattern — same approach used in v0.3.0 (shell → runtime extraction)

### Integration Points
- `runtime.ts` handleReq() — relay pool and cache logic extracted to service
- `runtime.ts` handleEvent() kind 29001 case — signer logic extracted to service
- `RuntimeHooks` interface — remove relayPool, cache; keep auth, crypto, etc.
- `@napplet/services` — add createRelayPoolService(), createCacheService(), createSignerService(), createCoordinatedRelay()

</code_context>

<specifics>
## Specific Ideas

- The composite service pattern (`createCoordinatedRelay`) is the reference implementation's answer to "how do I combine relay pool and cache?" Shell implementors who use different stacks just write their own.
- The incremental migration mirrors v0.3.0's shell → runtime extraction — proven pattern for this codebase.
- Signer is the simplest migration (request/response pattern, no subscription lifecycle). Start there as proof of concept.

</specifics>

<deferred>
## Deferred Ideas

- High-level service helpers (beyond createCoordinatedRelay) — future convenience layer
- Removing old RuntimeHooks interfaces entirely — follow-up cleanup after verification
- Hotkey forwarding as a service — small, could go either way
- DM sending as a service — currently an optional hook

</deferred>
