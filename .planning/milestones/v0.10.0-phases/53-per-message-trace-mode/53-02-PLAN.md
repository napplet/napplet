# Plan 53-02: Color State Trace Mode Support

<objective>
Extend the color-state module to support a 4th persistence mode called 'trace'. When trace mode is active, persistent state recording is suppressed (no-op) and edge color queries return null (resting state), since trace mode drives edge colors entirely through per-message animations managed by trace-animator.ts.
</objective>

## Tasks

<task id="53-02-01">
<title>Add 'trace' to PersistenceMode type and update recording/query logic</title>
<read_first>
- apps/demo/src/color-state.ts (full file — understand PersistenceMode type, recordEdgeColor, getEdgeColor, setPersistenceMode functions)
</read_first>
<action>
Make the following changes to `apps/demo/src/color-state.ts`:

1. **Extend the PersistenceMode type** — change the type alias from:
```typescript
export type PersistenceMode = 'rolling' | 'decay' | 'last-message';
```
to:
```typescript
export type PersistenceMode = 'rolling' | 'decay' | 'last-message' | 'trace';
```

2. **Make recordEdgeColor a no-op when trace mode is active** — add an early return at the top of the `recordEdgeColor` function body, before the `const key = ...` line:
```typescript
// Trace mode: persistent state is not accumulated — animations are ephemeral
if (_mode === 'trace') return;
```

3. **Return null from getEdgeColor when trace mode is active** — add a case to the switch statement in `getEdgeColor`:
```typescript
case 'trace':
  return null; // Trace mode has no persistent state — edges driven by animations
```

These changes ensure that when the user switches to trace mode:
- No persistent state accumulates (recordEdgeColor is a no-op)
- All edges report null (resting) for their persistent color
- The existing `setPersistenceMode` already clears accumulated state on mode switch
- The change notification still fires on mode switch, causing the persistent renderer to clear all edge/node colors
</action>

</task>
