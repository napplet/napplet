# Plan 52-01: Service Toggle Infrastructure in shell-host.ts

<objective>
Add a persistent handler reference store and a public toggleService() API to shell-host.ts so that services can be disabled (unregistered from runtime) and re-enabled (re-registered from stored reference) without losing the handler object. Expose service-enabled state for other UI modules to query.
</objective>

Then modify the wrapped `registerService` inside `bootShell()` (around line 500-504) to also store the handler in the permanent map:

Change from:
```typescript

    originalRegisterService(name, handler);
  };
```

This ensures every handler registered through the demo's wrapped API is permanently retained.

<action>
In `apps/demo/src/shell-host.ts`, add two new exported functions after the existing `toggleBlock()` function (after line 638):

```typescript
/**
 * Enable or disable a service. When disabled, the service handler is unregistered
 * from the runtime (messages to it will fail). When re-enabled, the stored handler
 * reference is re-registered. Changes take effect on the very next message.
 */
export function toggleService(name: string, enabled: boolean): void {
  if (enabled) {
    const handler = serviceHandlerStore.get(name);
    if (!handler) {
      console.warn('[service] toggleService: no stored handler for', name);
      return;
    }
    disabledServices.delete(name);
    relay.runtime.registerService(name, handler);
    console.log(`[service] ENABLED ${name}`);
  } else {
    disabledServices.add(name);
    relay.runtime.unregisterService(name);
    // Re-add to demoServiceNames so it still appears in topology (just disabled)
    demoServiceNames.add(name);
    console.log(`[service] DISABLED ${name}`);
  }
}

/**
 * Check if a service is currently enabled (registered with the runtime).
 */
export function isServiceEnabled(name: string): boolean {
  return !disabledServices.has(name);
}
```

Note the important detail on disable: after `unregisterService` removes the name from `demoServiceNames` (via the existing wrapper at line 506-508), we immediately re-add it. This way the service still appears in the topology node list but is visually marked as disabled. The wrapped `unregisterService` calls `demoServiceNames.delete(name)` before `originalUnregisterService(name)`, so we re-add after.

Also add `toggleService` and `isServiceEnabled` to the module's public API exports. They are already exported via the `export function` keyword.
</action>
