<objective>
Finalize Phase 114 by emitting the three downstream artifacts that consume the validated `resolvedSchema` from plans 114-01 and 114-02:

1. **Manifest tag** (VITE-04): add `['config', JSON.stringify(schema)]` to the kind 35128 manifest event tags.
2. **aggregateHash participation** (VITE-05): include a synthetic `config:schema` x-tag-style line so schema changes propagate into aggregateHash (and therefore into the napplet's storage scope).
3. **Meta tag injection** (VITE-06): inject `<meta name="napplet-config-schema" content="{escaped JSON}">` into the built `index.html` head so the shim from Phase 113 can read the schema synchronously at iframe load.

Purpose: This is the output-side plumbing. Plans 114-01 and 114-02 produced a validated `resolvedSchema` closure variable; this plan makes that schema visible to every downstream consumer:
- The NIP-5A manifest (kind 35128) — shell-side relays, signature validators, and napplet-ACL gatekeepers can see the declared schema alongside the napplet's hashes + requires.
- The `aggregateHash` — storage scoping is keyed on `(dTag, aggregateHash)`, so including schema bytes in the hash means a schema change is effectively a schema version bump that forks storage (aligning with the NUB-CONFIG `$version` potentiality without prescribing migration).
- The runtime meta tag — `installConfigShim()` from phase 113 reads `<meta name="napplet-config-schema">` at install time to populate `window.napplet.config.schema` synchronously. Without this injection, manifest-declared schemas would not surface runtime-side.

### 2. Extend `closeBundle` — add synthetic aggregateHash path AND append manifest config tag

Inside the existing `closeBundle` body, locate the `// Compute aggregate hash` step. The pattern is:

```ts
// Walk dist/ and compute per-file SHA-256
const files = walkDir(distPath);
const xTags: Array<[string, string]> = [];
for (const relativePath of files) {
  if (relativePath === '.nip5a-manifest.json') continue;
  const fullPath = path.join(distPath, relativePath);
  const hash = sha256File(fullPath);
  xTags.push([hash, relativePath]);
}

// Compute aggregate hash
const aggregateHash = computeAggregateHash(xTags);
```

Insert the synthetic schema-hash line between the dist-walk loop and the `computeAggregateHash` call:

```diff
 const files = walkDir(distPath);
 const xTags: Array<[string, string]> = [];
 for (const relativePath of files) {
   if (relativePath === '.nip5a-manifest.json') continue;
   const fullPath = path.join(distPath, relativePath);
   const hash = sha256File(fullPath);
   xTags.push([hash, relativePath]);
 }

+// NUB-CONFIG: include the config schema bytes in aggregateHash via a synthetic
+// path. The colon in 'config:schema' ensures no collision with any real file
+// path (dist/ relative paths use platform separators, not colons). Any schema
+// change flips the schema-bytes sha and therefore flips aggregateHash — which
+// in turn re-scopes the napplet's storage per NIP-5D (dTag, aggregateHash)
+// keying. Schemas thus implicitly version their own storage without requiring
+// `$version` cooperation.
+if (resolvedSchema !== null) {
+  const schemaBytes = JSON.stringify(resolvedSchema);
+  const schemaHash = crypto.createHash('sha256').update(schemaBytes).digest('hex');
+  xTags.push([schemaHash, 'config:schema']);
+}
+
 // Compute aggregate hash
 const aggregateHash = computeAggregateHash(xTags);
```
