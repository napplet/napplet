<objective>
Extend `@napplet/vite-plugin` with a three-path schema discovery system: inline `configSchema` option, `config.schema.json` convention file, or `napplet.config.ts` export. Land the option type, discovery helper, and `configResolved`-hook integration that populates a plugin-instance `resolvedSchema` variable consumed by downstream plans (validation in 114-02, manifest + meta emission in 114-03).

Purpose: This is the input-side plumbing for Phase 114. All subsequent work in this phase (guards, manifest tag, aggregateHash, meta tag injection) depends on the plugin having a resolved schema value to operate on. Split into its own plan because:
1. The existing plugin source is ~200 LOC with no discovery logic â€” adding three discovery paths + dynamic import of TS/JS config files is a non-trivial addition on its own.
2. Failure modes here (parse error, import error, missing file vs. present-but-unreadable) are distinct from the schema-validity failure modes of plan 02.
3. Keeping discovery isolated makes the `resolvedSchema` interface the sole contract between 114-01 and 114-02/03 â€” those later plans read a plain `JSONSchema7 | null` and don't care where it came from.

### 4. Add `discoverConfigSchema` helper ABOVE the `nip5aManifest` factory

Place this function between `computeAggregateHash` and the JSDoc block on `nip5aManifest`. Structure:

```ts
/**
 * Three-path schema discovery. Returns the resolved schema + source name, or
 * null/null when no schema is declared anywhere.
 *
 * Precedence (each step is strict â€” later steps run only when the earlier
 * step yielded no schema):
 *   1. `options.configSchema` is an object -> use it directly (source: 'inline option').
 *   2. `options.configSchema` is a string -> resolve relative to `root`, read+parse as JSON (source: 'inline option: <path>').
 *   3. `config.schema.json` exists at `root` -> read+parse as JSON (source: 'config.schema.json').
 *   4. `napplet.config.ts` / `.js` / `.mjs` exists at `root` -> dynamic import, read `configSchema` named export (source: 'napplet.config.<ext>').
 *   5. None -> return { schema: null, source: null }.
 *
 * Parse / import / missing-export failures throw with an explanatory message.
 * The "file does not exist" case is NOT an error â€” it advances to the next path.
 */
async function discoverConfigSchema(
  options: Nip5aManifestOptions,
  root: string,
): Promise<{ schema: JSONSchema7 | null; source: string | null }> {
  // Step 1 + 2: inline option
  if (options.configSchema !== undefined) {
    if (typeof options.configSchema === 'object') {
      return { schema: options.configSchema, source: 'inline option' };
    }
    if (typeof options.configSchema === 'string') {
      const p = path.isAbsolute(options.configSchema)
        ? options.configSchema
        : path.resolve(root, options.configSchema);
      if (!fs.existsSync(p)) {
        throw new Error(
          `[nip5a-manifest] configSchema path does not exist: ${p}`,
        );
      }
      try {
        const raw = fs.readFileSync(p, 'utf-8');
        return { schema: JSON.parse(raw) as JSONSchema7, source: `inline option: ${p}` };
      } catch (err) {
        throw new Error(
          `[nip5a-manifest] failed to parse configSchema file ${p}: ${(err as Error).message}`,
        );
      }
    }
  }

  // Step 3: convention file
  const conventionPath = path.resolve(root, 'config.schema.json');
  if (fs.existsSync(conventionPath)) {
    try {
      const raw = fs.readFileSync(conventionPath, 'utf-8');
      return { schema: JSON.parse(raw) as JSONSchema7, source: 'config.schema.json' };
    } catch (err) {
      throw new Error(
        `[nip5a-manifest] failed to parse config.schema.json at ${conventionPath}: ${(err as Error).message}`,
      );
    }
  }

  // Step 4: napplet.config.* dynamic import
  for (const ext of ['ts', 'js', 'mjs'] as const) {
    const cfgPath = path.resolve(root, `napplet.config.${ext}`);
    if (!fs.existsSync(cfgPath)) continue;
    try {
      // Convert to file:// URL for ESM dynamic import on Windows + Linux
      const url = new URL(`file://${cfgPath}`).href;
      const mod = await import(url);
      const schema = (mod.configSchema ?? mod.default?.configSchema) as JSONSchema7 | undefined;
      if (schema === undefined) {
        throw new Error(
          `[nip5a-manifest] napplet.config.${ext} at ${cfgPath} does not export \`configSchema\` (neither as a named export nor on the default export)`,
        );
      }
      return { schema, source: `napplet.config.${ext}` };
    } catch (err) {
      throw new Error(
        `[nip5a-manifest] failed to load napplet.config.${ext} at ${cfgPath}: ${(err as Error).message}`,
      );
    }
  }

  // Step 5: nothing found â€” silent, backward compatible
  return { schema: null, source: null };
}
```

Notes:
- The `napplet.config.ts` import path has a known caveat: Vite-plugin runtime is Node (not processed by Vite), so a raw `.ts` file cannot be dynamically imported without a loader. For v1, if the user places a `.ts` config file, Node's default ESM loader throws and we surface a message telling them to use `.js` or `.mjs`. Do NOT shell out to `tsx`/`esbuild` for this â€” keep the dependency surface zero. The `.ts` branch is present for forward compat (Node 22+ with `--experimental-strip-types`), but the documented path is `napplet.config.js` or `.mjs`.
- `mod.configSchema ?? mod.default?.configSchema` allows both `export const configSchema = {...}` and `export default { configSchema: {...} }` shapes. README in phase 116 will document the preferred form.
