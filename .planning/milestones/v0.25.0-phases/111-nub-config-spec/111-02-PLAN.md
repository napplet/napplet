    ### Core Subset

    Describe what every conformant shell MUST support. Structure as three explicit lists:

    **Supported types (Core Subset):**
    - `object` (top-level only — schema root MUST be `{ "type": "object", "properties": { ... } }`)
    - `string`
    - `number`
    - `integer`
    - `boolean`
    - `array` of primitives (homogeneous only — tuple-typed arrays `items: [a, b]` are NOT in Core Subset)
    - Nested `object` (bounded depth — recommend maximum depth of 4)

    **Supported keywords (Core Subset):**
    - `type`
    - `properties`
    - `required` (array form)
    - `items` (homogeneous schema only)
    - `additionalProperties` (see override below)
    - `default`
    - `title`
    - `description`
    - `enum`
    - `enumDescriptions` (parallel-array hint matching `enum`)

    **Supported constraints (Core Subset):**
    - `minimum`, `maximum` (numbers/integers)
    - `minLength`, `maxLength` (strings)
    - `minItems`, `maxItems` (arrays)

    **`additionalProperties` override:**
    Shells MUST treat `additionalProperties` as `false` at the top-level schema object when the napplet does not specify it explicitly. This overrides the JSON Schema draft-07 default of `true` for NUB-CONFIG scope. Rationale: persisted settings must not accrete unknown properties silently.

    **Default-resolution rule (deterministic):**
    1. If a persisted value exists AND validates, use it.
    2. Else if the property has its own `default`, use that.
    3. Else if an ancestor `default` object supplies a value for this property, use the ancestor's value.
    4. Else the property is `undefined` in the delivered `config.values`.

    ### Standardized Extensions (Potentialities)

    These are opt-in annotations napplets MAY declare. Shells MAY act on them; shells MAY ignore them. Shells that do not recognize an extension MUST treat it as opaque metadata (no validation failure).

    | Keyword | Applies to | Shell MAY |
    |---------|------------|-----------|
    | `x-napplet-secret` (boolean) | any string property | mask input, suppress logging, decline to render default, use platform keychain |
    | `x-napplet-section` (string) | any property | group properties under the declared section heading |
    | `x-napplet-order` (non-negative number) | any property | sort within section ascending; ties broken alphabetically |
    | `deprecationMessage` (string) | any property | render next to the field as a deprecation notice |
    | `markdownDescription` (string) | any property | render as markdown (fall back to plain text) |

    ### `$version` Potentiality

    A schema MAY carry a top-level `$version` integer field. Shells MAY use `$version` to drive cross-hash migration when the napplet's `aggregateHash` changes; shells MAY ignore it and treat each hash as a fresh scope. NUB-CONFIG does NOT prescribe migration strategy — migration is a shell concern, and `$version` is a signal, not a contract.

    Napplets MUST NOT receive values violating the currently-registered schema at `config.values` delivery time, regardless of how the shell reconciles older persisted values.

    ### Exclusions (Not in v1 Core Subset)

    The following are explicitly excluded from NUB-CONFIG v1:

    - **`pattern` keyword** — excluded due to ReDoS (Regular Expression Denial of Service) risk. JavaScript's built-in `RegExp` engine is backtracking-based; a napplet-declared pattern such as `^(a|a)*$` against a 31-character input can consume tens of seconds of CPU on the shell's main thread. See CVE-2025-69873 (ajv validator) for a concrete real-world precedent of this class of vulnerability in JSON Schema validators. Readmission requires a shell-safe regex strategy (Worker + timeout, RE2-WASM, or equivalent) to be specified in a future revision of this NUB.
    - **`$ref`** — ALL forms forbidden, including same-document (`#/definitions/foo`), external URL (`https://...`), filesystem (`file://`), and cross-document relative. Rationale: schemas are self-contained; `$ref` resolution is a security surface (exfiltration, DoS via slow URLs, TOCTOU) and a renderer-complexity explosion.
    - **`definitions` / `$defs`** — follows from the $ref exclusion.
    - **`oneOf`, `anyOf`, `allOf`, `not`** — combinatorial schemas are out of Core Subset v1; shells MAY support as Extended Subset but napplets MUST NOT rely on them.
    - **`if` / `then` / `else`** — conditional schemas are out of scope.
    - **`patternProperties`, `propertyNames`, `dependencies`, `dependentSchemas`, `unevaluatedProperties`** — draft 2019-09+ features out of scope.
    - **Tuple-typed arrays** (`items: [schemaA, schemaB]`) — homogeneous arrays only in Core Subset.
    - **Recursive `$ref` of any kind** — forbidden by the $ref exclusion but called out explicitly.
    - **Expression-valued defaults** (`"$now()"`, `"$random()"`, etc.) — `default` values are literal JSON per JSON Schema semantics; napplets needing runtime-generated defaults MUST handle the `undefined` case in their own code.

    ### Format as Hint Only

    The JSON Schema `format` keyword (`email`, `uri`, `date`, `date-time`, `color`, `ipv4`, `ipv6`, etc.) is annotative in NUB-CONFIG: shells MAY use `format` to pick an input widget but MUST NOT reject a value solely because it fails the `format`. Napplets requiring strict validation MUST use `enum`, `minLength`, `maxLength`, `minimum`, or `maximum` — NOT `format`.

    ### Depth Limit

    Shells MUST reject schemas nesting more than 4 levels deep at `config.registerSchema` time, returning `config.registerSchema.result` with `ok: false` and `code: "schema-too-deep"`. Napplets MUST NOT assume arbitrary nesting is supported.
