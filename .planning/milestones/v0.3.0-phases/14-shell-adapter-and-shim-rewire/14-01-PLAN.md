## Objective

Transform `@napplet/shell` from a monolithic protocol engine into a thin browser adapter that delegates all protocol logic to `@napplet/runtime`. The `createShellBridge()` function must internally call `createRuntime(adaptHooks(hooks))` while preserving the existing `ShellHooks` interface for backwards compatibility. Browser-specific modules (origin-registry, state-proxy, manifest-cache, audio-manager) remain in shell.

## Context

Currently `shell-bridge.ts` contains ~650 lines of protocol logic: message dispatch, AUTH handshake, subscription management, replay detection, event buffering, signer proxying, shell commands, audio commands. After Phase 13, all protocol logic lives in `@napplet/runtime` via `createRuntime(hooks: RuntimeHooks)`. Phase 14 rewires the shell to delegate to runtime while keeping browser-specific code (Window references, localStorage, CustomEvents, postMessage) in the shell package.

The key challenge is writing `adaptHooks()` — a function that converts `ShellHooks` (browser-facing, Window-oriented) into `RuntimeHooks` (environment-agnostic, windowId-oriented). The adapter bridges:
- `originRegistry` (Window <-> windowId) for message routing
- `state-proxy` (localStorage-backed) for napp state operations
- `manifest-cache` (localStorage-backed) for napp update detection
- `audio-manager` (Web Audio API) for audio source tracking

<action>
Rewrite `packages/shell/src/shell-bridge.ts` to be a thin adapter:

1. Remove ALL protocol logic (message dispatch, AUTH handling, subscription management, replay detection, event buffering, signer handling, shell commands, audio commands, hotkey handling). This all now lives in `@napplet/runtime`.

2. The new `createShellBridge(hooks: ShellHooks): ShellBridge` must:

```typescript
import { createRuntime } from '@napplet/runtime';
import type { Runtime } from '@napplet/runtime';
import { adaptHooks } from './hooks-adapter.js';
import { originRegistry } from './origin-registry.js';

export function createShellBridge(hooks: ShellHooks): ShellBridge {
  const runtimeHooks = adaptHooks(hooks, { originRegistry, manifestCache, aclStore, audioManager, nappKeyRegistry });
  const runtime: Runtime = createRuntime(runtimeHooks);

  return {
    handleMessage(event: MessageEvent): void {
      const sourceWindow = event.source as Window | null;
      if (!sourceWindow) return;
      const windowId = originRegistry.getWindowId(sourceWindow);
      if (!windowId) return;
      const msg = event.data;
      if (!Array.isArray(msg) || msg.length < 2) return;
      // Delegate to runtime — runtime handles verb dispatch, AUTH, queueing
      runtime.handleMessage(windowId, msg);
    },

    sendChallenge(windowId: string): void {
      runtime.sendChallenge(windowId);
    },

    injectEvent(topic: string, payload: unknown): void {
      runtime.injectEvent(topic, payload);
    },

    destroy(): void {
      runtime.destroy();
    },

    registerConsentHandler(handler): void {
      runtime.registerConsentHandler(handler);
    },
  };
}
```

3. The `ShellBridge` interface remains UNCHANGED — same 5 methods (handleMessage, sendChallenge, injectEvent, destroy, registerConsentHandler).

4. The `handleMessage` method is the ONE place that bridges browser and runtime: it extracts `sourceWindow` from `MessageEvent`, looks up `windowId` via `originRegistry`, then delegates `(windowId, msg)` to the runtime. This is the only browser-specific logic.

5. Keep the `ShellBridge` interface export and JSDoc comments.
</action>
