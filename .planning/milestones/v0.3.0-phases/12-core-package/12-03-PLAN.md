# Plan 12-03: Rewire @napplet/shim to Import from @napplet/core

<objective>
Replace all local protocol type and constant definitions in @napplet/shim with imports from @napplet/core. Delete or reduce shim's local types.ts to contain only shim-specific types (if any). The shim package declares @napplet/core as a dependency. The shim's public API (subscribe, publish, query, emit, on, nappStorage, NostrEvent, NostrFilter) remains unchanged.
</objective>

<context>
After Plan 12-01, @napplet/core exists and exports all protocol types and constants.

Currently, `packages/shim/src/types.ts` contains:
- NostrEvent interface (duplicate of shell's)
- NostrFilter interface (duplicate of shell's)
- BusKind constant (duplicate)
- BusKindValue type (duplicate)
- AUTH_KIND constant (duplicate)
- SHELL_BRIDGE_URI constant (duplicate)
- PROTOCOL_VERSION constant (duplicate)

All of these are exact duplicates of what @napplet/core now provides.

The shim has no shim-specific types in types.ts — everything there is protocol-level. So types.ts can become a pure re-export file (or be deleted with imports changed to @napplet/core directly in each consumer file).

Strategy: Replace types.ts with re-exports from @napplet/core. This minimizes changes to other shim source files since they all import from './types.js'.
</context>

<task id="12-03-02" title="Replace shim's types.ts with re-exports from @napplet/core">
<read_first>
- packages/shim/src/types.ts
- packages/core/src/index.ts
</read_first>
<action>
Replace the entire contents of `packages/shim/src/types.ts` with re-exports from @napplet/core:

```ts
/**
 * @napplet/shim — Protocol types re-exported from @napplet/core.
 *
 * Shim previously maintained its own copies of these types.
 * They now live in @napplet/core as the single source of truth.
 */

export type { NostrEvent, NostrFilter } from '@napplet/core';
export { BusKind, AUTH_KIND, SHELL_BRIDGE_URI, PROTOCOL_VERSION } from '@napplet/core';
export type { BusKindValue } from '@napplet/core';
```

Key points:
- All items that were previously defined locally are now re-exported from @napplet/core
- Uses `export type` for pure type exports (verbatimModuleSyntax compliance)
- This approach avoids changing every file that imports from './types.js' — they continue to import from the same path, but the definitions now come from core
- File is reduced from ~52 lines to ~10 lines
</action>
