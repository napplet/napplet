- Wire format: `["REGISTER", { "dTag": "<napplet_type>", "claimedHash": "<aggregate_hash_or_empty>" }]`
- The napplet MUST send REGISTER before any other message

**New subsection 2.3 Identity Delegation** (insert after REGISTER) — Document the IDENTITY verb:
- Direction: shell -> napplet
- Wire format: `["IDENTITY", { "pubkey": "<hex>", "privkey": "<hex>", "dTag": "<type>", "aggregateHash": "<hash>" }]`
- The shell MUST derive the keypair deterministically: `seed = HMAC-SHA256(shellSecret, dTag + aggregateHash)`, then `pubkey = schnorr.getPublicKey(seed)`
- `shellSecret` is a 32-byte random value generated once per shell and persisted across sessions
- Same napplet type + version + shell instance = same keypair
- The shell MUST send IDENTITY before sending AUTH challenge
- If the shell cannot derive a keypair (no shell secret persistence), it MUST send `["NOTICE", "shell secret not available — cannot derive keypair"]`
- Add note: the `aggregateHash` in IDENTITY may differ from `claimedHash` in REGISTER if the shell overrides based on verification

**Renumber existing 2.2 to 2.4 (Sequence)** — Update the 7-step flow:
- New step 1: Napplet sends REGISTER
- New step 2: Shell derives keypair, sends IDENTITY
- New step 3: Shell sends AUTH challenge (same as old step 1)
- New step 4: Napplet signs AUTH with delegated key (replaces old step 2 + 3). Update the event example to note the event is signed with the delegated private key (not an ephemeral one). Keep the kind 22242 event format the same.
- Update step 5 (old step 4, shell verifies): Add sub-step that shell verifies AUTH pubkey matches the pubkey sent in IDENTITY. If mismatch: `"auth-required: pubkey mismatch — AUTH must use delegated key from IDENTITY"`.
- Steps 5-7 (OK, Mandatory tags, Post-AUTH replay) stay mostly the same

**Update old 2.5 (dTag Derivation, renumbered)** — Replace the `parseInt(pubkey.slice(0, 8), 16).toString(36) + nappletType` formula. The dTag now comes directly from the REGISTER payload (napplet declares its own type). The old pubkey-based derivation is removed.

**Add subsection on Instance GUIDs** (brief, after dTag) — The shell MAY assign a persistent GUID per iframe slot, surviving page reloads. This is an implementation concern, not a wire format requirement. The GUID is not part of the postMessage protocol.

**Add subsection on Hash Verification** (before Compatibility Check) — During REGISTER handling, if the shell has a hash verifier, it SHOULD compute the aggregate hash from fetched napplet files and compare to claimedHash. On mismatch, the shell SHOULD display a user warning. Verification results SHOULD be cached by `dTag:aggregateHash` — recomputation only needed when the hash changes. This is a SHOULD, not MUST — dev-mode shells may skip verification.

<action>
Edit SPEC.md Section 14 with these specific changes:

**14.1 Threat Model** — Add a fifth assumption: "Delegated napplet keys are confined to the shell-napplet postMessage channel and MUST NOT appear on external relays"

**14.2 Security Layers** — Update layer 2 (NIP-42 AUTH): Change from "One-time Schnorr signature verification establishes napplet identity" to mention that the napplet signs AUTH with a shell-delegated deterministic keypair (not an ephemeral one). The shell verifies the signature matches the pubkey it derived and sent in IDENTITY.

Update layer 8 (Storage scoping): Change `(pubkey, dTag, aggregateHash)` to `(dTag, aggregateHash)`. Update text to match new scoping model.

Add a new layer 9: **Delegated key confinement:** Events signed by delegated napplet keys are blocked from relay publishing. The shell maintains a set of delegated pubkeys and rejects EVENT messages signed with delegated keys for non-internal kinds. Only the user's signer (NIP-07/NIP-46) produces events published to external relays. The publishing flow is: napplet requests publish -> shell checks ACL -> shell asks user's signer to sign -> shell publishes to relays. The delegated key is nowhere in this chain.

**New subsection 14.4 Delegated Key Threat Analysis** — Add after 14.3:

1. **Key exfiltration via fetch():** A malicious napplet could exfiltrate its delegated private key via `fetch()` to its serving origin (the only network access allowed by iframe sandbox). This is harmless because: (a) no relay has ever seen that pubkey — events signed with it would be meaningless, (b) the shell blocks relay publishing with delegated keys, (c) the key is deterministic so the napplet always gets the same one.

2. **Shell secret compromise:** If the `shellSecret` is leaked, an attacker can derive all napplet keypairs for that shell. Impact is limited because delegated keys cannot publish to relays. The attacker could impersonate napplets to the shell via a crafted iframe, but this requires access to the user's browser.

3. **Cross-shell key isolation:** Different shell instances (different `shellSecret`) derive different keypairs for the same napplet. A napplet's delegated key from shell A is meaningless on shell B.

4. **Derivation collision resistance:** HMAC-SHA256 with a 32-byte random key provides 256-bit collision resistance. The probability of two different `(dTag, aggregateHash)` pairs producing the same keypair is ~3.7 * 10^-39, which is negligible.
</action>
